# Firmware-locked fan control on Linux: what ventd can and cannot do

**Bottom line: of the four locked-down hardware classes ventd will encounter, only one (Supermicro AST BMCs) is genuinely *not* locked once you understand it; the other three are stuck at varying degrees, with one sanctioned profile-only API (HPE iLO5/6), one effectively-permanent vendor refusal (Dell iDRAC9 ≥ 3.34 on 14G), and one firmware-side enforcement on specific GPU SKUs.** This report tells ventd users honestly which group their hardware falls into, what commands actually work, and when to reach for an inline PWM controller, a firmware downgrade, or simple acceptance of telemetry-only mode. Every recommendation here is sourced from primary documentation, vendor KB articles, or reproduced forum reports from 2023–2026; uncertainty is flagged inline rather than papered over. The pattern across all four cases is the same: vendors who lock fans do so in the BMC/firmware *server-side*, where no Linux-side cleverness can route around them — the only real escape hatches are pre-existing profile APIs, hardware substitution, or firmware versions that predate the lock.

## 1. Dell PowerEdge 14G with iDRAC9 ≥ 3.34.34.34 — locked, no real escape

**Workaround status: NONE for direct fan control. PARTIAL for noise reduction via sanctioned thermal-profile knobs.**

Dell engineering deliberately removed `ipmitool raw 0x30 0x30 0x01/0x02` access in **iDRAC9 firmware 3.34.34.34** (late 2019). A Dell representative on the linux-poweredge mailing list and Dell Community confirmed: *"In 14g Dell changed the way we did the thermal algorithms… Going forward access is not going to be allowed as it affects the thermal algorithms and cooling the system."* A Dell escalation manager separately told user Vincent Cojot Dell *"will never put the feature back because it shouldn't have been there in the first place."* The block has held through every release in the 4.x, 5.x, 6.x, and 7.x series up to **iDRAC9 7.30.10.50 (2026)**, and **iDRAC10 on 17G (R670/R770) shipping in 2025–2026 carries the same posture**. The block is enforced server-side regardless of caller privilege — promoting the IPMI user to "Administrator+OEM" still returns `rsp=0xd4 Insufficient privilege level`.

**What still works on iDRAC9 ≥ 3.34** is the documented `system.thermalsettings` group, surfaced identically via racadm, the iDRAC web UI, and Redfish `PATCH /redfish/v1/Managers/System.Embedded.1/Attributes`. The single highest-value command for homelab users with non-Dell HBAs/NICs/GPUs is **`racadm set system.thermalsettings.ThirdPartyPCIFanResponse 0`** (or its IPMI raw equivalent `ipmitool raw 0x30 0xce 0x00 0x16 0x05 0x00 0x00 0x00 0x05 0x00 0x01 0x00 0x00`), which prevents iDRAC from ramping fans to ~70-100% just because it sees an unrecognized PCIe card. Beyond that, ventd can switch `ThermalProfile` between Default/Maximum Performance/Minimum Power/Sound Cap, raise (never lower) `MinimumFanSpeed`, raise `FanSpeedOffset` (Low/High/Medium/Max — *offset upward only*), and tighten `AirExhaustTemp` or `MaximumPCIeInletTemperatureLimit`. **There is no Redfish endpoint, racadm command, or undocumented OEM IPMI command anywhere in the public reverse-engineering record that sets per-fan PWM or pushes fans below Dell's calculated floor on 14G ≥ 3.34.**

**Firmware downgrade is effectively impossible on any 14G server in service today.** The last working version is iDRAC9 **3.30.30.30**. Dell erected two downgrade barriers: a Hardware Root-of-Trust certificate barrier introduced by **4.40.40.00 in February 2021** (every server shipped from Dell after June 2021 had this baked in), and a bootloader barrier introduced June 2024 once iDRAC9 reached **7.00.00.172** that defeats even the documented two-step path through 4.40.10.00. Dell KB 000225924 explicitly states this carries forward in all newer releases. Even if a user finds a server old enough to downgrade, they take on **CVE-2019-3705 (unauthenticated stack-buffer-overflow RCE, CVSS 9.8), CVE-2019-3706 (unauthenticated web auth bypass, CVSS 9.8), and CVE-2019-3707 (unauthenticated WS-MAN auth bypass, CVSS 8.6)** — six years of missing patches. Anyone exposing iDRAC outside a strictly isolated management VLAN cannot run pre-3.34 firmware.

**Active community projects all hit the same wall.** The most popular, **tigerblue77/Dell_iDRAC_fan_controller_Docker** (441★, last release v1.11 in November 2025), explicitly documents in its README that it works only on ≤ 3.30.30.30. Forks (xsmarty/idrac9_fan_controller, OCT0PUSCRIME, barnhill, Konata09, jsenecal/ipmi-fan-control, Darkhand81/idrac_fan_control) are all `ipmitool raw 0x30 0x30 …` wrappers and inherit the block. The ipmitool upstream issue #30 requesting Dell OEM fan commands has been open since 2014 with no merge; even if merged it would not change the server-side enforcement. **No upstream relief is in flight** — there are no Dell-specific patches in linux-hwmon, freeipmi, OpenBMC, or any tree, because the block is in the closed iDRAC firmware where no community project has access.

**Inline PWM controllers are not a documented working pattern on 14G.** Dell uses 6-pin proprietary fan connectors, and iDRAC continuously compares tach RPM to a model keyed to chassis SKU + populated PCIe + DIMM + drive backplane + CPU. Out-of-range tach reads produce "Fan N RPM" critical events and drive *other* fans to 100% as compensation; multiple missing tach signals can inhibit POST. **Zero confirmed long-running 14G deployments with Corsair Commander Pro, Aquacomputer Quadro, or NZXT Grid+ substitutions surfaced in any 2023–2026 forum search.** Tach-spoofing with a 555 timer or donor fan is well-known PC enthusiast practice but unvalidated on 14G.

**iDRAC8 / 13G is the escape hatch that still works.** On R630/R730/R730xd/T630 with iDRAC8 firmware up to and including 2.85.85.85 / 2.86.x in 2026, `ipmitool raw 0x30 0x30 0x01 0x00` and `0x30 0x30 0x02 0xff <pct>` still function — confirmed by 2024–2025 reports on the Proxmox forum, Dell Community, and Reddit. The yinheli/ipmi-fan-control Rust controller targets exactly this hardware. ventd should explicitly recommend that 14G owners who care about quiet either accept the floor, downgrade to 13G, or move workloads to non-Dell hardware.

> **What ventd should tell 14G owners:** "Your iDRAC blocks all userspace fan-PWM control. ventd will set ThirdPartyPCIFanResponse=Disabled, choose your thermal profile, and report telemetry. There is no path to lower fans below Dell's floor: no IPMI raw, no Redfish, no racadm secret. Downgrading is blocked by hardware Root-of-Trust on essentially all in-service 14G hardware, and would re-expose unpatched CVE-2019-3705. If quiet matters, use 13G/iDRAC8."

## 2. HPE ProLiant Gen10/Gen10+/Gen11 with iLO5/iLO6 — profile-only, by design

**Workaround status: NONE for direct PWM. PARTIAL via four-bucket `ThermalConfiguration` profile and a system-wide minimum-percent floor.**

HPE iLO5 and iLO6 expose **no writable Fan reading, target RPM, or per-fan PWM property to user space, ever, on any firmware version**. This is confirmed against the HPE-published Redfish schema, the iLO5/iLO6 changelogs, the iLO IPMI User Guide, and a Telegraf reproduction (issue #14488) showing `/redfish/v1/Chassis/1/Thermal` Fan members are read-only `CurrentReading`, `FanName`, `Status`, `Units`. **No working IPMI raw OEM netfn on Gen10+ has been published in any forum 2023–2026.** Unlike Dell, HPE never had a working IPMI raw fan command and removed it — they simply never exposed one on iLO5+.

The four sanctioned knobs ventd can operate are: **`Oem.Hpe.ThermalConfiguration`** on `/redfish/v1/Chassis/1/Thermal` with no reboot, taking the enum **`OptimalCooling | IncreasedCooling | MaximumCooling | EnhancedCooling`** (note value-name mismatch with the BIOS `ThermalConfig` attribute which uses `MaxCooling` / `EnhancedCPUCooling`); **`Oem.Hpe.FanPercentMinimum`** as a system-wide percent floor that can only *raise* fans; per-sensor user temperature thresholds (which only generate IML log entries, not curve changes); and the dangerous-but-real `Oem.Hpe.FallBackDisabled` per-sensor toggle for unrecognized 3rd-party card sensors (HPE warns this risks "loss of thermal protection"). HPE documents `IncreasedCooling` specifically as the lever for "third-party storage controllers" and "thermal issues that cannot be resolved" — this is the closest equivalent to Dell's `ThirdPartyPCIFanResponse=0`, but as a system-wide profile rather than a per-slot override. **iLO5/6 has no per-PCIe-slot cooling override at all** — a real architectural gap vs. iDRAC9.

The single best non-mod intervention for any Gen8–Gen11 ProLiant with high idle fan speed is **installing HPE AMS (`amsd` or `hp-ams` from the HPE Linux SDR repo)** so iLO can read drive temperatures via the OS. Without AMS, drives in AHCI/non-VROC mode produce no temp data and fans stay elevated. Reports show idle fan drops from 43% → 6–8% on DL20 Gen10/Gen11 and similar reductions on DL360/DL380 Gen10. **HPE Community thread for DL360 Gen11 (BIOS 2.44, 2025) and the Rocky Linux ML110 Gen11 amsd guide both treat amsd installation as the single highest-impact action** — ventd's HPE documentation should make this Step 0 before any profile automation.

**Gen8/Gen9 with iLO4 is the one HPE platform where direct fan control exists**, but only via patched firmware. The canonical project is **kendallgoto/ilo4_unlock** (520★), which builds patched iLO4 v2.77 with the SSH `fan p N min/max <0–255>` command unlocked (HPE removed the fan utilities in v2.78+, so v2.77 is the latest unlockable). Tested on DL360p G8, DL380p G8/G9, DL80 G9, SL4540 G8, ML350 G9. **That-Guy-Jack/HP-ILO-Fan-Control** and **alex3025/ilo-fans-controller** wrap this with autofan systemd services and Docker UIs. All explicitly state Gen10+ with iLO5/6/7 are not supported and "probably never will be" — **no homebrew patched iLO5 or iLO6 firmware exists on GitHub as of April 2026**, because iLO5/6 use stronger boot verification on different SoCs. ventd can offer opt-in support for the patched-iLO4 path but must warn about voiding HPE support, requiring the iLO security override switch, brick risk, and the documented quirk that chaining `fan` commands with `;` crashes iLO until full power-cycle.

**Inline PWM controllers and Noctua fan replacements on Gen10+ trigger iLO "Fan N failure" alarms** because iLO validates HPE-specific tach signatures via the proprietary 6/7-pin connectors. Forum reports of Noctua swaps on ML30 Gen10 / ProLiant boxes show iLO either marking fans failed and ramping survivors to 100%, or — once BIOS fan warnings are explicitly disabled — bypassing fan control entirely (degraded telemetry, no thermal response). There is no widely-reproduced tach-spoof recipe for Gen10+ comparable to the Dell pulldown trick. **GPU-aware cooling on HPE only works for HPE-branded GPU SKUs with the matching High Performance Heatsink Kit + Temperature Fan Kit** (e.g. SKUs 826706-B21, 867810-B21); non-HPE H100/A100/MI300 PCIe cards in DL380 Gen10/Gen11 are treated as third-party PCIe and run fans high regardless of actual GPU temp. **Apollo chassis (XL270d, Apollo 2000/6500) defer to chassis-manager fan policy that supersedes per-tray iLO** — ventd should not assume per-server control on Apollo trays.

Firmware downgrade on iLO5/iLO6 is generally allowed via the UpdateService API and SUM, but **no version ever exposed more fan control**, so this is not a productive path on Gen10+.

> **What ventd should tell HPE Gen10+ owners:** "iLO5/6 lets ventd switch among four cooling profiles and raise (never lower) a percent floor — that's the entire writable fan API. Install amsd first; this alone often halves idle fan speed. For per-card hot-running PCIe, use IncreasedCooling globally — there is no per-slot knob. On Gen8/Gen9 only, opt-in patched iLO4 via kendallgoto/ilo4_unlock unlocks real per-fan control at the cost of an HPE security-switch override and warranty implications."

## 3. Supermicro X10/X11/X12/X13/H11/H12/H13 with AST2400/2500/2600 BMC — almost everything is controllable

**Workaround status: FULL on virtually every consumer/server board, once you understand the model. The "FANA/FANB/CPU_OPT firmware-locked" claim that motivated this research is largely a myth.**

The canonical raw command pattern, byte-validated across forums and the smfc reference implementation, is: **`ipmitool raw 0x30 0x45 0x01 0x01`** to enter Full mode, then **`ipmitool raw 0x30 0x70 0x66 0x01 <zone> <pwm>`** with `<zone>` = `0x00` (CPU/system, FAN1–FANn) or `0x01` (peripheral/HDD, FANA/FANB), and `<pwm>` as decimal 0–100 or hex 0x00–0x64. Read-back is `0x30 0x70 0x66 0x00 <zone>`. **Byte order is identical across X10/X11/X12/X13/H11/H12/H13** — petersulyok's smfc (612 commits, 341★, actively maintained, Python with full platform abstraction) confirms the AST2600 boards (X13SAE-F, H13SSL-N) accept the same command pattern. The four fan modes are `0x00` Standard, `0x01` Full, `0x02` Optimal, `0x04` Heavy IO; **only Full mode preserves user PWM long-term**, as Standard and Optimal silently revert writes within seconds-to-minutes per the smfc README and a decade of reproductions.

The "firmware-managed channel" myth dissolves on inspection. On every common single-CPU board (X11SCH, X11SSH-F/L, X11SSL, X11DPi/DPH dual-CPU, X12STH, X13SAE-F, H11SSL-i, H11DSi-NT, H12SSL family, H13SSL-N, M12SWA-TF), all populated headers (FAN1–FANn + FANA + FANB) are user-controllable in Full mode via the two-zone command — **there is no board in the AST2400/2500/2600 family known to firmware-lock any header that the BMC monitors**. Two genuine cases of "I can't control these" are commonly mistaken for vendor lockdown: **multi-zone chassis/blade boards (X10DRU-i+, X10QBi, GPU 4027GR-TR, X12 BigTwin)** expose 3–4 zones and require probing `0x30 0x70 0x66 0x01 0x00..0x04 0x32` to discover them (smfc Issue #20 documents this), and **PSU-internal fans on Supermicro PWS-* PSUs are managed entirely by the PSU MCU and are never reachable via 0x30 0x70 0x66** — they appear via PMBus tach-read only. On chassis boards FANA/FANB are simply chassis fans on Zone 1, *not* the PSU.

The single most common Supermicro pain point — **"I set PWM low and fans roar to 100% a few seconds later"** — is the lower-fan-threshold trap. Supermicro BMCs assert Lower-Non-Recoverable / Lower-Critical when any monitored fan crosses the threshold and rescue-spin every fan. The canonical PigLover STH fix is **`ipmitool sensor thresh FAN1 lower 0 100 200`** (LNR=0, LCR=100, LNC=200 RPM), repeated for every populated header. **AST2600 (X13/H13) honors only Lower Critical** — LNR/LNC writes are silently ignored (smfc Issue #33). Thresholds reset on BMC firmware update on every board, and on cold BMC reset on some firmware revisions, so ventd must re-apply on its own startup. Upper non-recoverable thresholds can also trigger 100% ramp on some firmware; setting `upper <NC> <CR> <NR>` to fan-max-plus-headroom is good defensive practice.

**The major caveats and unknowns are concentrated at the edges**: PWM range above 0x64 (100) overflows or wraps on most boards (luckylinux verified on X9DRi-LN4F+; ventd should clamp at 100 defensively). Mode `0x03` returns inconsistent semantics across firmwares (sometimes "PUE", sometimes "Heavy IO"). **X14/H14, the post-AST2600 generation with Supermicro's new SAA/SSM management stack, has unknown raw-command compatibility — Supermicro explicitly removed IPMICFG/SMCIPMITool support for X14/H14**, so smfc currently lists "compatibility under investigation." ventd should test before assuming the byte sequence works on X14/H14 and likely needs a Redfish path for that generation.

**Native host-side Linux fan control is not possible** on any Supermicro board with stock BMC firmware: `aspeed-pwm-tacho` and `aspeed-g6-pwm-tach` are mainline kernel drivers, but they run on the BMC's ARM core (for OpenBMC), not the host x86. The OpenBMC port to X11SSH-F is partial (Hardenedvault 2024, 3mdeb ZarhusBMC 2025); the official Supermicro-OpenBMC GitHub fork lists no X12/X13/H12/H13 production targets. Ecosystem tools that work today: **smfc** (de-facto standard), **luckylinux/supermicro-fan-control**, **jasongaunt/Supermicro-Fan-Control** (older), **khorton/nas_fan_control PID Perl**, and TrueNAS community **spinpid2.sh**. Inline PWM controllers (Aquacomputer Quadro/Octo) are mechanically straightforward: feed BMC a tach-in-spec signal from a donor fan or the Quadro's own tach output, set lower thresholds to zero on those headers, run real fans off the controller — though no canonical megathread documenting this on Supermicro surfaced, so flag as plausible-extrapolation rather than confirmed.

> **What ventd should tell Supermicro owners:** "Almost every header on your board is controllable via the two-zone command pattern in Full mode — including the FANA/FANB you may have heard are locked. The thing that actually breaks naive scripts is the lower-RPM threshold trap; ventd will set thresholds to safe values on startup and re-apply after BMC firmware updates. If a fan header doesn't respond, you're on a multi-zone chassis board — probe zones 0–4. PSU-internal fans are not BMC-controlled on any Supermicro board. X14/H14 support is provisional pending Redfish migration."

## 4. GPUs: vBIOS, drivers, and the chassis-fan distinction

**Workaround status varies by vendor and SKU; the critical clarification ventd users need is GPU-onboard fans vs. chassis fans.** GPU-onboard fans (the fans physically attached to the GPU PCB) are wired directly to the GPU's own controller IC or SMU — **an external inline PWM controller cannot intercept these without physical board modification**. For vBIOS-locked or driver-locked SKUs the only options are vBIOS flash, card replacement, or accepting the firmware default curve. Chassis fans are entirely separate and live on the motherboard or BMC; ventd controls those normally via NCT6775/AST hwmon while reading GPU temp via vendor SMI tools. For server/datacenter cards (H100/H200/B100/B200, L40, MI300, Gaudi), the card has no fan or has firmware-locked fan and chassis fans are controlled by the **server BMC**, which gets GPU temps out-of-band via **NVIDIA's SMBPBI at I2C 0x4F**, AMD's MCTP-over-SMBus equivalent, or Intel's similar path. These flow into Redfish `/Chassis/.../Thermal` on the BMC; ventd cannot reach them from the host.

**NVIDIA consumer (RTX 30/40/50 — Ampere/Ada/Blackwell)** under driver 555+ has a working NVML path: `nvmlDeviceSetFanControlPolicy()` (added in 520) and `nvmlDeviceSetFanSpeed_v2()` (added in 515) succeed on most cards, confirmed by LACT 0.7.x logs against driver 580 on RTX 4090-class hardware. **NVML works headless, on Wayland, and on systems without Xorg installed** — this is the recommended path going forward. The X11+Coolbits fallback still works on Ada Lovelace and Blackwell under drivers 555–580 with `nvidia-xconfig --cool-bits=28` (4=fan, 28=fan+OC+overvolt) followed by `nvidia-settings -a "[gpu:0]/GPUFanControlState=1" -a "[fan:0]/GPUTargetFanSpeed=80"`, requires `DISPLAY=:0`, an active X server, and root-privileged X (`needs_root_rights = yes` since driver 465). **`nvidia-smi` has no fan-set flag** — confirmed by years of identical user complaints; do not waste user time looking for hidden options. The Xvfb-headless pattern still functions for legacy setups but is brittle (the NVIDIA driver wants something it recognizes as a connected display, so users often resort to HDMI/DP dummy plugs). **GSP firmware (default in 555+, mandatory in 560+) does not block NVML fan control**, but **vBIOS-side locks on specific SKUs do silently no-op the API** — the canonical example is the RTX 2070 Super FE on drivers 550–580 (NVIDIA developer forum thread 347334). `nvmlDeviceSetDefaultFanSpeed_v2` is documented-but-broken (developer forum 214430): calling it doesn't actually resume the auto curve.

**NVIDIA datacenter cards** are uniformly outside ventd's direct control. H100 PCIe, H200 PCIe, A100 PCIe, L40, L40S, L4, RTX PRO 6000 Blackwell Server Edition: card-onboard blower (where present) is GPU-firmware-managed, **not exposed to nvidia-smi for set**. SMBPBI at I2C 0x4F is the BMC-side ingress. SXM modules (H100/H200/B100/B200 SXM) have no card fan at all. RTX 6000 Ada and RTX 5000 Ada workstation cards do respond to NVML/nvidia-settings (same user-space stack as consumer Ada). DGX H100/H200 expose chassis thermal via `/redfish/v1/Chassis/.../Thermal` on the BMC (NVIDIA Redfish API supplement).

**AMD consumer** is the most user-friendly. Set kernel cmdline `amdgpu.ppfeaturemask=$(default | 0x4000)` to enable the OD bit (`PP_OVERDRIVE_MASK`, bit 14), then write `/sys/class/drm/card0/device/hwmon/hwmon*/pwm1_enable` and `pwm1`. **Linux 6.14+ marks the kernel tainted "out-of-spec"** when overdrive is on (commit b472b8d829c1); ventd should report this honestly to users. Per-generation: **RX 6000 (RDNA2)** uses the classic single-`pwm1` interface and works on any modern kernel. **RX 7000 (RDNA3)** removed single-PWM manual mode and replaced it with the SMU fan-curve sysfs (`gpu_od/fan_*`); LACT and CoreCtrl 1.4+ use this. Zero-RPM disable requires kernel ≥ 6.13. **RX 9000 (RDNA4) requires kernel ≥ 6.14, linux-firmware late-Feb 2025+**, and 6.15+ for full fan-speed reporting and zero-RPM toggle (Phoronix, multiple Lawstorant patches landed for 6.15). Datacenter Instinct MI200/MI250/MI300/MI325 are passively cooled OAM modules — `amd-smi`/`rocm-smi` fan APIs return N/A. W7900/W7800 workstation cards use the RDNA3 SMU curve.

**Intel is read-only as of April 2026.** Arc A-series fan-RPM reporting landed in **Linux 6.12** for i915 — fan1_input only, **no pwm1 write path**. Battlemage (Arc B580/B570) under the `xe` driver has the same read-only state through kernel 6.18. **Intel uploaded fan-control firmware blobs for Battlemage to linux-firmware.git in early 2026**, but this is firmware/driver-internal control; userspace still cannot write. No public patch series for user-writable PWM on i915/xe surfaced. Intel Data Center GPU Max (Ponte Vecchio), Falcon Shores, and Habana Gaudi 2/3 are passively cooled — chassis fans only, BMC-managed.

**Active GPU fan-control projects** in priority order: **LACT** (ilya-zlobintsev, the de-facto cross-vendor tool, AMD via sysfs and NVIDIA via NVML, headless support, RDNA4 fan curves landed in 0.7.2, current 0.8.5+); **CoreCtrl 1.4+** (AMD, Qt5/C++20 GUI, RDNA3 curves added in March 2024); **NVFD** (Infinirc, NVML-based ncurses TUI for X11/Wayland/headless); **amdgpu_top** (Umio-Yasuno, monitor-only, no fan write); **amdgpu-profile-switcher** (Umio-Yasuno, per-process profile switcher useful as a ventd integration pattern); **foucault/nvfancontrol** (still authoritative for Coolbits values though X-leaning); **Cippo95/nvidia-fan-control** (script-based, useful pattern). The Spect3r/Nvidia-GPU-Fan-Control project the original query asked about did not surface as actively maintained in 2024–2026 searches — flag as likely abandoned. **Nouveau on Blackwell** (Ben Skeggs' GH100/GB10x/GB20x patch series for Linux 6.16) delegates reclocking to GSP and adds no manual fan PWM; manual fan control on NVIDIA remains a proprietary-driver/NVML-only path. The **NOVA Rust driver** has skeleton work in 6.15–6.17 with no fan-control surface yet.

> **What ventd should tell GPU owners:** "Use NVML for NVIDIA on driver 555+; it works on Wayland and headless. Coolbits+X11 still works on Ada and Blackwell as a fallback. Some cards (e.g., RTX 2070 Super FE) accept the call but the vBIOS silently ignores it — there is no userspace override; replace the card or flash the vBIOS. For AMD set ppfeaturemask|=0x4000, but Linux 6.14+ taints the kernel; on RDNA3+ you must use the curve interface, not single-PWM. RDNA4 needs kernel 6.15+. Intel is read-only on Linux as of 2026 — monitor only. Datacenter GPUs are BMC/Redfish territory; ventd does not touch GPU-onboard fans there."

## Cross-cutting patterns and ventd's recommendation matrix

Three patterns repeat across all four cases and should shape ventd's user messages.

**Vendor lockdowns are server-side firmware enforcement, not client-side. Userspace cleverness cannot route around them.** Dell's iDRAC9 ≥ 3.34 block, HPE iLO5/6's read-only Fan resources, NVIDIA's vBIOS GPU-fan curves on locked SKUs, and AMD's pre-OD-bit safety gate are all enforced where the user has no execution surface. The reverse-engineering record for 2023–2026 contains **zero published bypasses** for any of these — and the absence is meaningful, because all four cases have large, motivated communities (homelab, mining, AI tinkerers) who would have published a bypass if one existed.

**Firmware downgrade is real on Supermicro, partial on HPE, and effectively impossible on Dell 14G.** Supermicro freely permits BMC firmware downgrade via the web UI, SUM, or `ipmitool hpm`. HPE iLO5/6 permits downgrade via UpdateService and SUM but pre-FIPS/CNSA mode (HPE bug a00136386 documents the exception); the catch is that no HPE firmware ever exposed more fan control than current. Dell iDRAC9 on 14G has a Hardware Root-of-Trust barrier from February 2021 plus a bootloader barrier from June 2024 that together block all in-service systems from reaching pre-3.34 firmware. **Even where downgrade is technically feasible, the security cost is real**: pre-3.34 iDRAC9 carries unauthenticated-RCE CVE-2019-3705. ventd should never recommend downgrade without the CVE disclosure attached.

**Inline PWM controllers (Aquacomputer Quadro/Octo, Corsair Commander Pro, NZXT Grid+) are the right "your hardware is locked, accept it" answer for chassis fans on Supermicro and consumer/workstation systems, are documented-failure-prone on HPE Gen10+ and Dell 14G because the BMC validates tach signatures, and are simply not applicable to GPU-onboard fans.** ventd's "your hardware is locked" message should branch on platform: for Supermicro multi-zone chassis or PSU fans, recommend an inline controller plus a tach donor and threshold-zeroed BMC headers; for HPE Gen10+/Dell 14G, recommend either accepting the sanctioned profile API (HPE), accepting iDRAC's floor (Dell), or migrating to 13G/Supermicro/older HPE Gen9 with patched iLO4. For locked NVIDIA consumer cards, recommend accepting the vBIOS curve or replacing the card. For datacenter GPUs, recommend operating ventd against the **server BMC's Redfish thermal endpoint** rather than the host GPU stack.

**Upstream/in-flight kernel work moves the needle in only two places over the next 6–12 months.** AMD RDNA4 fan-control polish continues (Lawstorant patches landing through 6.15-6.16, zero-RPM toggle, gpu_metrics fallback when hwmon absent). Intel Battlemage may eventually receive a user-writable PWM path but no public patch series exists as of April 2026 — ventd should not promise this. Nothing relevant is in flight for Dell, HPE, or any closed-BMC fan control: the OpenBMC X11SSH-F port for Supermicro (Hardenedvault, 3mdeb ZarhusBMC) is the only realistic path to host-side native fan control on any of these classes, and only for AST2400-era boards. For ventd's release horizon, the picture is functionally static.

## Conclusion: a four-line decision tree for the ventd amendment

The honest message ventd should embed in its hardware-detection paths is this. **On Dell PowerEdge 14G with iDRAC9 ≥ 3.34**, ventd is limited to disabling third-party PCIe response, choosing a thermal profile, and raising — never lowering — the fan floor; users who want quiet must move to 13G/iDRAC8 hardware. **On HPE ProLiant Gen10/Gen10+/Gen11 with iLO5/iLO6**, ventd installs amsd as Step 0 and then chooses among four ThermalConfiguration profiles plus a system-wide percent floor; there is no per-fan or per-slot control and never has been. **On Supermicro AST2400/2500/2600 boards**, ventd is free — the two-zone command works on every common board, the only real engineering task is setting lower fan thresholds defensively and clamping PWM at 100; the "FANA/FANB locked" claim is a misreading of multi-zone chassis boards and PSU-internal fans. **On GPUs**, ventd controls AMD consumer via sysfs after enabling ppfeaturemask|=0x4000 (with kernel taint disclosure on 6.14+), controls NVIDIA consumer via NVML on driver 555+ (with vBIOS-lock acceptance for the silent-no-op SKUs), reads Intel as monitor-only, and on datacenter cards uses BMC Redfish not the host GPU stack. The novel insight from cross-vendor reading is that the four "lockdowns" are not the same kind of problem: Dell is a deliberate vendor refusal that 2026's most popular community project (tigerblue77, last release Nov 2025) explicitly cannot solve; HPE is a pre-existing API gap framed as a "by design" position; Supermicro is mostly a documentation problem the community already solved; and GPU locks are SKU-specific firmware policy that has no userspace bypass and never will.
