# R28 — Fan Control Failure Modes — Agent F: Distro Bug Trackers

**Scope.** Distro-side bug tracker mining: Ubuntu Launchpad, Debian BTS, Fedora/Red Hat Bugzilla, openSUSE/SUSE Bugzilla, Arch Linux GitLab, NixOS GitHub. Goal: identify per-distro fan-control quirks where ventd needs distro-aware behaviour (modprobe.d conventions, DKMS rebuild ordering, signed-module flow, hwmon enumeration assumptions, suspend/resume restart hooks, secure-boot lockdown blocking sensors-detect, etc.).

**Method.** Searched each tracker for fan-control / hwmon / lm-sensors / fancontrol / thinkpad_acpi / nct6775 / nct6683 / it87 / dkms / thermald / mokutil keywords. Followed the most relevant ~20 bugs to extract status, fixed-in-version, root cause, distro-specific aspect. Sibling agents A-D-E mine the upstream / forum / kernel side; this file deliberately stays in distro-tracker territory.

**Time window.** Bugs from 2020 onward preferred (kernel + distro have moved on); a few older ones included where the underlying convention is unchanged and distro-specific.

**Confidence column.** H = bug is closed-fixed with a verifiable version stamp or backport. M = bug is open/confirmed but root cause and distro-specific aspect are clear. L = bug is invalid/EOL but reveals a workaround pattern still applicable.

**Trackers that resisted scraping.**
- bugzilla.suse.com search via Google returns mostly unrelated kernel-default bugs; component-filtered fan/hwmon results are sparse — openSUSE coverage in the table is thin and largely indirect (community / forum signals via ThinkWiki, lm-sensors upstream).
- bugzilla.opensuse.org keyword search behaves the same way.
- Some Red Hat Bugzilla entries returned 303 redirects — workaround was to refetch via search snippets. One entry below (1669564) was reconstructed from search excerpt + parallel kernel commit hint.

## Per-Distro Patterns ventd's Auto-Fix Endpoints Need to Dispatch On

**Ubuntu (Launchpad, by far the largest signal).** Two big patterns:
1. **HWE/GA kernel flavour split.** Every kernel switch (jammy → jammy-hwe-22.04 → noble) re-triggers DKMS rebuilds for any out-of-tree sensor module (it87, nct6687d, asus-wmi-sensors). Ubuntu's own dkms postinst ordering bug (#1681566) is fixed, but per-module FTBFS bugs keep reappearing. ventd's DKMS auto-fix should: detect HWE vs GA flavour, check `dkms status` against `uname -r`, and trigger `dkms autoinstall -k $(uname -r)` if a known sensor module is built only for the previous kernel.
2. **Suspend/resume restart.** The `/lib/systemd/system-sleep/fancontrol` hook is the canonical Ubuntu fix (#1882272 + the gating fix #1967432). On Ubuntu pre-Bionic 1:3.4.0-4ubuntu0.1 / pre-Focal 1:3.6.0-2ubuntu1.1 the hook is missing — ventd should drop in its own.

**Debian.** Same suspend/resume pattern but solved differently: bookworm/trixie's `lm-sensors 1:3.6.2-1` ships `Restart=on-failure` on `fancontrol.service` (#664141). On Debian, ventd should *not* install a competing sleep hook — let systemd restart handle it. Also: Debian's `pwmconfig` regenerates configs that don't always parse for `fancontrol.service` (#893234), so ventd's own config writer should be authoritative, not `pwmconfig`.

**Fedora.** Two distinct distro-specific paths:
1. **Secure Boot + kernel_lockdown** silently breaks `sensors-detect` ISA probing (#1842611). On any SB-enabled Fedora, ventd must skip the ISA-probe phase and rely on PCI/ACPI enumeration only — and it should *tell the user* this is happening, because the failure mode otherwise looks like "no chip detected".
2. **akmods + RPM Fusion signing flow** for any DKMS module on SB. Different binary (`akmods`, `kmodgenca`, `mokutil --import /etc/pki/akmods/certs/public_key.der`) than Ubuntu (`dkms`, `update-secureboot-policy`). The install pipeline currently treats all "DKMS distros" the same — this is wrong and the highest-priority gotcha.

**openSUSE / SUSE.** Coverage thinnest. The signal is: openSUSE lacks the apt-postinst kernel-update trigger that Ubuntu/Debian have, so DKMS modules get orphaned silently after `zypper dup` (bug 1184671 generalises). Convention for module options is `/etc/modprobe.d/` (same as Debian-family) but initramfs refresh is `dracut -f`, not `update-initramfs`. ventd needs an `initramfs_tool` switch keyed on distro.

**Arch.** Distro tracker has nearly no fan-control bugs because Arch users go straight to forums/AUR. The one canonical Arch failure mode is gcc/kernel skew on `nvidia-dkms` rebuilds (#7). Arch's `/etc/modprobe.d/` convention matches Debian. No HWE/GA flavour problem (rolling). ventd's Arch path is the simplest: drop config in `/etc/modprobe.d/`, `mkinitcpio -P` to refresh.

**NixOS.** Genuinely different from everything else. The big three:
1. **`/etc/modprobe.d/*.conf` written at runtime is ignored** — must be part of the system closure. ventd cannot drop a config file imperatively; it must emit a Nix module fragment for the user to import.
2. **`/etc/sensors.d/*.conf` same problem** (issue #437830).
3. **No suspend/resume hook shipped** (the `fancontrol.nix` module documents the failure but doesn't fix it).

Net: NixOS needs its own ventd backend that emits Nix expressions, not files.

**Conclusion on support burden:** Ubuntu's tracker has the most absolute traffic *and* the cleanest fix stamps — it's where ventd can rely on "fixed in version X" gating. Fedora's pattern is "closed EOL" before fix lands, so ventd should detect Fedora N and N-1 and warn rather than wait for a future fix. NixOS has the highest *unhandled* burden — open issues sit stale because the ecosystem expects the user to write Nix.

## Distro-Specific Gotchas the Install Pipeline Doesn't Yet Handle

1. **Fedora SB+lockdown silently breaks ISA chip detection.** Pipeline currently runs `sensors-detect` style probes — these will return empty on any SB-enabled Fedora and the user gets no signal why.
2. **Fedora's akmods/RPM-Fusion signing path** is divergent from Ubuntu's mokutil/dkms path. If ventd ships a kernel module (or instructs DKMS to), the signing step needs distro dispatch.
3. **NixOS `/etc/modprobe.d/` and `/etc/sensors.d/` are not authoritative.** Any imperative config drop is ignored; ventd must emit a Nix module fragment instead.
4. **NixOS lacks a suspend/resume restart hook for fancontrol.** Pipeline must install one (or use `Restart=on-failure` style override) when ventd runs on NixOS.
5. **openSUSE lacks the apt-postinst → DKMS-rebuild trigger chain.** After `zypper dup` of `kernel-default`, sensor DKMS modules silently go missing. Pipeline needs a post-kernel-upgrade check independent of the package manager.
6. **Ubuntu HWE↔GA kernel switches** orphan DKMS-built sensor modules. Pipeline must detect kernel-flavour switch (compare `uname -r` vs previous boot) and trigger rebuild — not wait for the next reboot to discover modules missing.
7. **Debian lm-sensors <1:3.6.2-1** doesn't have `Restart=on-failure`; pipeline should detect old version and either drop a sleep hook or set the systemd override itself.
8. **Vendor-ID `force=1` modprobe pattern** for nct6683/nct6685D/nct6687 is needed cross-distro. Pipeline currently does not auto-write `options nct6683 force=1`; it should, after detecting Nuvoton EC via DMI but absent hwmon sysfs entry.
9. **`thermal_zone` returning `-61` (EBADMSG)** on Ubuntu 24.04 Acer/NVIDIA combo — pipeline should treat this as "kernel doesn't trust this zone" and not as an empty-temp state.
10. **Dell-SKU-level quirks** (e.g. OptiPlex XE4 mask thermald). ventd's quirk DB must support DMI SKU number, not just vendor+model.
