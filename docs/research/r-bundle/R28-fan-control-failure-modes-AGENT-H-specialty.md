# R28 — Fan-Control Failure Modes (Agent H, Specialty Hardware)

**Scope.** Mining the long-tail hardware that the other six R28 agents don't cover well: Apple Silicon under Asahi Linux, Steam Deck (LCD + OLED) under SteamOS / Bazzite / mainline, server-class boards (Supermicro, Dell, HPE, Gigabyte server, ASRock Rack) on TrueNAS / Unraid / Proxmox, GPU server platforms (DGX), Intel Mac (`applesmc` legacy), HEDT / Threadripper PRO workstations on WRX80 / WRX90, ARM SBCs (Raspberry Pi 5, Orange Pi, Rock Pi, Khadas, RISC-V VisionFive), and unusual mini-PCs (Beelink, Minisforum, Geekom, Topton). Goal: structured, operator-actionable table of niche fan-control failure modes ventd will hit on this hardware.

**Method.** Parallel WebSearch / WebFetch across vendor docs (Asahi Linux docs, kernel.org hwmon docs, HPE iLO 6 user guide), GitHub repos (philipl/steamdeck-kernel-driver, evlaV/jupiter-fan-control, petersulyok/smfc, frankcrawford/it87, Joshua-Riek/ubuntu-rockchip, Fred78290/nct6687d, passiveEndeavour/it5570-fan), forum threads (TrueNAS, Unraid, Proxmox, Level1Techs, RVspace, Raspberry Pi forums, ServeTheHome, asahilinux Mastodon), and blog posts with confirmed-working configs (b3n.org Supermicro, kovasky.me iDRAC, jeffgeerling RISC-V, blog.tmm.cx WRX80, blog.pcfe.net MS-01).

**Caveats and coverage gaps.**
- Community knowledge skews toward hardware that has Linux enthusiasts. Mac Studio / MacBook Pro under Asahi has thorough docs because the Asahi project owns the driver; OEM mini-PC ECs (Geekom, AceMagic, GMKtec) are nearly undocumented because the EC firmware is proprietary and the user base runs Windows.
- Some sources (TrueNAS forums) require login for full thread bodies; rows reflect public excerpts only.
- M4 Mac series rows are speculative — the m1n1 bootloader is currently blocked on M4 (Apple changed the boot architecture); fan-control support is not yet in upstream macsmc-hwmon as of May 2026.
- HPE iLO licensing tiers are confirmed via HPE docs but the precise feature gates per-firmware are not exhaustively tested across all Gen10/11 SKUs.
- Steam Deck OLED ("Galileo") sensor map vs LCD ("Jupiter") differs but the upstream patches in the philipl tree handle both transparently; failure modes are biased toward the older firmware.
- RISC-V coverage is preliminary — VisionFive 2 is the main board with deployed Linux fan control, but the broader RISC-V SBC ecosystem (StarFive JH7110, SpacemiT K1, T-Head TH1520) lacks consistent hwmon driver support and rows are correspondingly thin.

## Summary — strategic categorisation

### Platforms where Linux fan control is "solved" (easy wins for ventd)
- **Apple Silicon under Asahi Linux** — `macsmc-hwmon` is upstream (kernel ≥6.5); `fan_control=1` opt-in module parameter is a one-line config on Fedora Asahi Remix or Asahi Arch. ventd should ship a doctor card that recommends the modprobe.d snippet on detected Asahi systems. Works for: MacBook Pro 14"/16" M1/M2/M3 Pro+Max, Mac mini M1/M2/M3, Mac Studio M1/M2 Ultra. (M4 not yet supported — track for v0.6+.)
- **Steam Deck under Bazzite / SteamOS** — `steamdeck-hwmon` driver shipped via Valve kernel patches (mainline distros need DKMS); `jupiter-fan-control` daemon already does the heavy lifting. ventd's job is to detect-and-not-fight: recognise DMI `Jupiter` / `Galileo` and offer a "Steam Deck PID monitor" mode rather than displacing the daemon. BIOS F7A0131+ is required.
- **Supermicro X11/X12 IPMI** — well-documented, two-zone OEM raw command, BMC firmware doesn't fight manual writes if BIOS is set to Full Speed. RULE-IPMI-1 happy path covers this. ventd is shippable here today; threshold-tuning UX is the polish work.
- **Raspberry Pi 5 + official Active Cooler** — `cooling_fan` device-tree overlay handles binding; ventd just needs to drive `pwm1` via the thermal_zone framework. DT-fingerprint matcher (RULE-FINGERPRINT-06) handles enumeration.

### Platforms that are "vendor-blocked" (detect-and-explain)
- **HPE iLO Standard tier** — fan write requires iLO Advanced license. RULE-IPMI-3 already returns "iLO Advanced required". ventd doctor should add a redfish license probe and surface the upgrade path (or recommend Standard Plus / Advanced license trial).
- **Dell iDRAC ≥3.34.34.34** — Dell revoked manual fan control via 0x30 0x30 raw. Detect via `ipmitool mc info` firmware version comparison; refuse engagement and explain the firmware-version gate. Older iDRAC (≤3.30.30.30) still works under RULE-IPMI-2.
- **HP ProLiant Gen8/G9** — proprietary HPE drivers required; lm-sensors path will never work. ventd should match HP DMI sys_vendor and offer an "install hpasm" link.
- **NVIDIA DGX / HGX (H100, H200, A100)** — datacenter GPU firmware locks fan_speed write. nvidia-smi reads but cannot write. ventd's NVML probe should look for "H100"/"H200"/"A100"/"L40S" in device name and refuse write-engagement, leaving control to the chassis BMC.
- **AMD Instinct MI200/MI300X** — same datacenter pattern, firmware-locked. Refuse and explain.
- **OEM mini-PCs without Linux EC driver** — Beelink, GMKtec, AceMagic, Topton, GEEKOM, MINISFORUM (some models) ship proprietary EC firmware with no in-tree driver. The `it5570-fan` OOT module covers a subset; the rest are monitor-only until community DKMS lands. ventd should DMI-pattern-match these vendors and offer the doctor card "no Linux EC driver — monitor-only mode".
- **Fanless mini-PCs / passively-cooled SBCs** — Topton N100/N305 fanless, Khadas VIM4, MacBook Air M1/M2 — no PWM surface exists. Classify as `OutcomeMonitorOnly` with evidence "no_controllable_channels".
- **Threadripper PRO WRX80 + ASUS Pro WS WRX80E-SAGE** — NCT6798D detected but BMC owns fans. No Linux PWM surface. ventd refuses with "configure fans in BMC web UI".

### Platforms growing fast — track for ventd v0.6+
- **Apple Silicon M4 (Mac mini M4, MacBook Pro M4, iMac M4)** — Asahi blocked on m1n1 bootloader; Apple changed XNU exception/pagetable behaviour in M4. Once upstream lands (likely 2026 H2), ventd inherits the same `macsmc-hwmon` path with new `apple,t8132` device-tree IDs. RULE-FINGERPRINT-06 already handles this — only the catalog entry needs a row.
- **Threadripper PRO 7000 series (WRX90 chipset)** — BMC-routed fan control is becoming the norm on these workstation boards. ventd must defer to the BMC web for fan curves and provide read-only telemetry. Some WRX90 SKUs (ASRock Rack TRX50W) expose fan control via NCT6798D — chip-level fingerprinting required.
- **RISC-V SBCs (StarFive JH7110, SpacemiT K1, T-Head TH1520)** — VisionFive 2 is the mature platform; the broader ecosystem is rapidly maturing. As more boards add DT-binding for `pwm-fan` and thermal-zones, ventd can extend RULE-FINGERPRINT-06 catalog. Until then, monitor-only.
- **Steam Deck OLED (Galileo) and any successor** — sensor map differs from LCD; coupling.Snapshot / signature shards must not be shared cross-revision. Catalog entry needs DMI product_name dispatch.
- **Mac Studio M4 / M3 Ultra** — when these ship under Asahi, the multi-fan map (4 fans on Studio) needs verification. Per-chassis fan count comes from device-tree.
- **NVIDIA Blackwell B200 / B100** — datacenter pattern likely persists; verify `nvmlDeviceSetFanSpeed_v2` returns NOT_SUPPORTED. If consumer Blackwell (B5000-class) ships, expect rw_full or rw_quirk capability.
- **AMD RDNA 4 (Navi 48, RX 8000-series)** — RULE-EXPERIMENTAL-AMD-OVERDRIVE-04 already handles RDNA4 + kernel <6.15 refusal. Track kernel uplift.

### Cross-cutting recommendations for ventd
1. **DMI vendor heuristics for "no Linux EC driver" mini-PCs** — pattern-match `sys_vendor` ∈ {Beelink, MINISFORUM, GEEKOM, GMK, AceMagic, Topton, CWWK, AOOSTAR} and surface a doctor card recommending it5570-fan / nct6687d / monitor-only based on chip fingerprint.
2. **BMC-firmware-version gate** — extend RULE-IPMI-2 / RULE-IPMI-3 to query firmware version via `ipmitool mc info` before engagement; refuse on Dell ≥3.34, refuse on HPE Standard.
3. **Two-controller-race detection** — detect concurrent userspace fan controllers (`jupiter-fan-control`, `mbpfan`, `fancontrol`, `thinkfan`, `nbfc`, Unraid `ipmifan`, NBFC, `asusctl`) via systemd unit names and refuse start. Existing PR-PREFLIGHT-CONFL-03 covers this.
4. **DT-fingerprint catalog entries** for ARM SBCs — Pi 5, Pi 4 PoE+ HAT, Orange Pi 5+, Rock 5B, VisionFive 2 — using the existing RULE-FINGERPRINT-06 / RULE-FINGERPRINT-07 matcher.
5. **Datacenter GPU detection** — extend NVML probe to recognise H100/H200/A100/L40S/MI200/MI300X by name and force OutcomeMonitorOnly. Avoids RULE-NVML-HELPER-EXIT-01 surfaces firing on every tick.
6. **macsmc-hwmon integration** — first-class doctor card explaining the `fan_control=1` opt-in. ventd never writes by default on Asahi without operator consent.
