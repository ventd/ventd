# R28 — Fan-Control Failure Modes (Agent D, StackExchange family)

**Scope.** Mining the StackExchange Q&A network — Ask Ubuntu, Unix & Linux SE, Super User, Server Fault — for upvote-validated symptom→fix pairs that ventd's recovery classifier can encode. Each row points at a real Q&A URL and captures the accepted-answer score (or top-voted answer where no green check exists, flagged in Notes).

**Method.** Tag-page crawl on `lm-sensors`, `fan`, `cpu-temperature`, `ipmi`, and per-vendor searches (ThinkPad, Dell, HP, ASUS, Gigabyte, Supermicro). HTML parsed for accepted-answer ID and the vote count immediately adjacent to the green check. Where the question has no accepted answer but a high-voted answer exists, the row uses that answer's score and flags `unaccepted` in Notes. WebFetch was blocked for direct StackExchange access; data was retrieved via curl with a desktop User-Agent.

**Filter applied.** Prefer accepted answers with score ≥ 5; include lower-scored rows when they cover a unique failure mode (otherwise the corpus would be dominated by the same handful of mega-popular answers).

## Summary

**Top-3 highest-upvoted answers (everyone hits this — top priority for shipping):**
1. **Q22108 "How to control fan speed?" — accepted answer 224 upvotes.** The canonical lm-sensors+pwmconfig+fancontrol pipeline. ventd's first-run wizard must mirror this flow's wording so users recognize it.
2. **Q53762 "How to use lm-sensors?" — accepted answer 117 upvotes.** First-time setup of `sensors-detect`. ventd onboarding should adopt the exact prompt/output language from this answer.
3. **Q328906 "Find fan speed and CPU temperature" — accepted answer 49 upvotes.** Recommends `s-tui` for terminal monitoring. ventd doctor mode could shell out to s-tui as a known-good readout comparison.

**Patterns with high upvotes but inconsistent fixes across boards (board-specific dispatch needed):**
- **`acpi_enforce_resources=lax`**: Appears as the fix in 6+ rows (it87, NCT6798, NCT6687, HP OMEN, generic). Works on most desktops; breaks `asus_atk0110` and others on some ASUS boards (Launchpad bug 953932 referenced repeatedly). ventd should not blanket-apply; it must check for known-broken-with-lax DMI strings first.
- **`force_id` for IT87 family**: Different boards need different IDs (0x8622 / 0x8628 / 0x8689 / 0x8688). ventd needs a chip-ID-prefix → working-force-id table indexed by motherboard DMI.
- **ASUS sensor access path**: nct6775 vs asus-ec-sensors vs asus_wmi_sensors — all whitelisted by motherboard model name. ventd must try each module in priority order rather than relying on `sensors-detect`.
- **AMDGPU PWM control**: Works on some kernels and dies (RDNA 1/2), not others (RDNA 3 stuck on auto). Needs per-GPU-architecture branch.

**High-view questions with no accepted answer (research-needed):**
- Q1474711 (NCT6798 fan readings missing on ASUS B560M/B660M boards) — exact same symptom as ventd should expect across most modern ASUS desktops.
- Q1515619 (B450 Aorus Elite fans show 0 RPM via IT8792) — a current-generation Gigabyte symptom.
- Q1553945 (RDNA 3 GPU fan control) — no software solution today; ventd should refuse rather than thrash.
- Q1556634 (ThinkPad E14 Gen 7 ACPI EC errors) — vendor BIOS bug; affects newest Lenovo line.
- Q648540 (no pwm-capable sensor modules on Huawei laptop) — affects budget laptops broadly.

**Cross-source confidence (Notes flag where applicable):**
- ThinkPad `fan_control=1` modprobe option: appears on AskUbuntu, ThinkWiki, ArchWiki, Gentoo wiki, Lenovo forums — uniformly high confidence.
- `acpi_enforce_resources=lax` for it87: AskUbuntu, ArchWiki, Linux Mint forums, Debian forums — high confidence on Gigabyte, partial on ASUS.
- Dell IPMI `raw 0x30 0x30` commands: ServerFault, joeplaa wiki, virtualbytes blog, GitHub repos — high confidence on PowerEdge ≤R740xd; broken on iDRAC ≥3.34.
- Supermicro `ipmitool sensor thresh` for low-RPM fans: ServerFault, Calvin Bui blog, haphazard.io, TrueNAS — high confidence on X10/X11.
