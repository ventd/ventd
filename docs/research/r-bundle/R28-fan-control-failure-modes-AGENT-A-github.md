# R28 ‚Äî Fan-Control Failure Modes Mined from OOT-Driver GitHub Issues (Agent A)

## Scope

This document catalogs operator-actionable failure modes harvested from the GitHub issue trackers of out-of-tree (OOT) Linux fan-control kernel drivers. The goal is to feed ventd's recovery classifier (`internal/recovery/classify.go`) with concrete detect+auto-fix rules so that users get a recovery card instead of an unexplained error.

### Repos surveyed

- `frankcrawford/it87` ‚Äî OOT IT8xxx fork (most active, ~40 closed issues reviewed)
- `Fred78290/nct6687d` ‚Äî Nuvoton NCT6687D fork (~50 closed issues reviewed)
- `electrified/asus-wmi-sensors` ‚Äî ASUS WMI sensor driver (was originally listed as `zylinski/asus-wmi-sensors`; that repo no longer exists, the active fork is `electrified`)
- `lm-sensors/lm-sensors` ‚Äî userspace tooling
- `nbfc-linux/nbfc-linux` ‚Äî laptop-EC fan-control daemon (the canonical "no SuperIO chip" path; added because so many laptop reports landed here)

### Repos missing or replaced

- `helge-fix/dell-bios-fan-control` ‚Äî does not exist; closest live repo is `TomFreudenberg/dell-bios-fan-control` and that repository has issues disabled. The Dell/SMM path is therefore not covered here; flag as gap.
- `zylinski/asus-wmi-sensors` ‚Äî does not exist; replaced with `electrified/asus-wmi-sensors`.
- `linuxhw/sensors` ‚Äî survey database, did not surface incident-style issues; skipped.

### Inclusion criteria

A row is only included if (a) the source issue has a kernel/dmesg/modprobe stamp ventd can grep for; (b) a comment confirms a fix that ventd could automate (modprobe option, blacklist drop-in, kernel cmdline, key-enrollment, sysfs write, config file); and (c) the fix does not require recompiling a kernel or hand-editing C source. Pure feature requests, board-bring-up requests, and "I gave up and bought a different board" threads are dropped.

### How to read the table

- **Class** is a proposed Go-style enum name. snake_case-readable so a maintainer can map it directly into a `FailureClass` constant.
- **Detection** is intended to be greppable from `dmesg` / `journalctl -k` / `modprobe` stderr, or to be a single sysfs / lsmod check. The exact phrase quoted in the source issue is preserved where possible.
- **Auto-fix** is the exact set of commands ventd would run. Where the fix needs a reboot (kernel cmdline), that is called out.
- **Confidence:**
  - `high` ‚Äî multiple users on multiple distros confirmed the same fix
  - `medium` ‚Äî single confirmed report, fix is mechanical and well-understood
  - `low` ‚Äî anecdotal / fix worked for one user but the underlying signal is weak
