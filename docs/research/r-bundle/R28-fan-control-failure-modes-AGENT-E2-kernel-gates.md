**E2 (this file)** turns every actionable workaround surfaced by R28 into a *kernel-version-gated dispatch decision* for ventd. Where a fix has landed mainline, the table records the kernel version, any stable-backport tags, and the precise pseudocode that ventd should evaluate. Where no fix exists yet, the row is filed as upstream-track and ventd's behavior on every kernel is documented.

## Methodology

1. Started from Agent E's 46 rows and re-verified every SHA via `gh api /repos/torvalds/linux/commits/<sha>` (5000-req/hr authenticated, no rate-limit pressure).
2. Mined `drivers/hwmon/{nct6775*,nct6683,it87,asus-ec-sensors,asus_wmi_ec_sensors,applesmc,dell-smm-hwmon,corsair-cpro,nzxt-kraken*,aquacomputer_d5next,surface_fan,macsmc,yogafan,cros_ec}`, `drivers/platform/x86/{thinkpad_acpi,asus-wmi,asus-nb-wmi,dell-smm-hwmon,alienware-wmi*,hp-wmi,acer-wmi,gigabyte-wmi,lg-laptop,msi-ec}`, `drivers/gpu/drm/amd/pm/swsmu/{smu11,smu13,smu14}`, `drivers/gpu/drm/amd/amdgpu/amdgpu_pm.c` for fan-related commits 2018-2026.
3. For each commit verified author/committer date + subject. Mapped author dates to the next mainline tag using the published Linux release calendar (kernelnewbies.org/LinuxVersions); when the surrounding merge tag was visible (`Merge tag 'hwmon-for-vX.Y'`), the assignment is **High** confidence.
4. Cross-referenced `Cc: stable@vger.kernel.org` tags via commit-message body to determine which LTS branches received the fix.
5. Where a workaround has *no* mainline fix, recorded the row as upstream-track with `Mainline kernel` = "none yet" and `Dispatch logic` = "fire on every kernel".

## Confidence rubric

- **High** — Commit verified, surrounding maintainer-tag merge visible OR `Cc: stable` plus dated backport set.
- **Medium** — Commit verified, version mapping inferred from author date alone; dispatch is correct but the precise version may be ±1 if the commit slipped a merge window.
- **Low** — Workaround documented but no mainline patch series found; ventd must always fire the workaround on this hardware.

## How `Dispatch logic` reads

The pseudocode is intentionally Go-shaped so it transcribes directly to ventd. `kernel.Version()` returns the running kernel as a `(major, minor)` tuple comparable via `>=`. `dmi.Match(...)` is a glob match against `/sys/class/dmi/id/{sys_vendor,product_name,board_name}`. `module.LoadedAndOlderThan(...)` checks the `vermagic` of an OOT DKMS module. `bmc.FirmwareVersion()` is `ipmitool mc info` parsed.

Where the row's mainline kernel column reads `none yet`, the dispatch is unconditional: "always do the workaround on this hardware" — no kernel gate.

## Summary

### Workarounds now obsolete on modern kernels (≥6.6 LTS)

The following classes can become *kernel-version-conditional* dispatches in ventd. On kernel ≥6.6 (current LTS) they need not fire at all; ventd should only execute the workaround on older kernels and instead recommend `apt install linux-image-6.6` etc. when the workaround would otherwise be needed.

- **`acpi_enforce_resources=lax` for it87 ACPI conflicts** (row 28) — replaced by `it87.ignore_resource_conflict=1` modprobe option since v6.2. **Eliminates a reboot.** This is the canonical case study from the Phoenix HIL discovery.
- **`force_id=0xd450` for NCT6799D** (row 9) — supported natively v6.4+; ventd should not recommend force_id on modern kernels.
- **`force_id=0x8628` for IT87952E** (row 30) — natively detected v6.4+.
- **`force=1` for many Dell laptops** (rows 46-50) — most major SKUs whitelisted by v6.10-v6.14; only G15 5510, G5 5505, OptiPlex 7080 are post-v6.18.
- **DMI quirk table for it87 pre-init crashes** (row 33) — eliminated v6.10 by self-detection rewrite.
- **65535 sentinel filter for ThinkPad fan2_input** (row 35) — fixed v6.2; remains harmless on newer kernels but no longer needed.
- **`ignore_resource_conflict` for nct6798 boards** (effectively row 17) — superseded by ASUS WMI path on v6.3+.
- **Dual-fan ThinkPad whitelisting** (row 36) — most P/X/T models added by v5.18; on kernel >= 5.18 ventd should expect both fans.

### Workarounds that still need to fire on every kernel (upstream-track)

These rows have **no mainline fix** as of v6.18 and ventd must always apply the workaround on the affected hardware:

- **AMD GPU OverDrive bit** for RDNA2/3/4 fan_curve writes (row 74) — by-design kernel taint gate.
- **ThinkPad `fan_control=1` modprobe option** (row 42) — by-design default.
- **Dell iDRAC9 ≥3.34.34.34 fan-control revocation** (row 110) — vendor-blocked.
- **HPE iLO 4 / Standard tier** (rows 111, 112) — vendor-blocked.
- **NVIDIA datacenter GPU fan_speed locked** (row 121) — vendor-blocked.
- **Apple Silicon SMC fan write** (rows 78-79) — Asahi declines to expose by default.
- **Aquacomputer pump config write** (row 84) — driver design.
- **NZXT Kraken3 rate-limit** (row 85) — firmware lock-up.
- **NixOS imperative modprobe.d** (row 123) — distro design.
- **Fedora SB+lockdown ISA probe** (row 124) — security design.
- **Supermicro LCR threshold tuning** (row 114), **0xff vs 0x64 H12 scale** (row 115), **Quanta D51B byte-spacing** (row 116), **Fujitsu non-Dell OEM** (row 117) — firmware variance, no fix.
- **ThinkPad DSDT re-assert curve** (row 103) — firmware behavior; ventd surfaces as BIOS-overridden monitor-only.
- **Dell SMM audio glitch / fan_state erratic** (rows 107-108) — firmware bug, no fix planned.

### Drivers with the most ongoing fix activity (long-tail signals)

1. **`drivers/hwmon/asus-ec-sensors.c`** — 30+ commits across 2025-2026; almost entirely "add board X" entries (Pro WS WRX90E-SAGE, X870-F GAMING, ROG STRIX X870E-H/X870-H, ROG STRIX X470-F, X470-I GAMING, X670E EXTREME, MAXIMUS X HERO, Pro WS TRX50-SAGE, Z270-A, ROG STRIX Z690-E, TUF GAMING X670E PLUS, ROG STRIX X670E-E, etc.). Predictable churn — ventd's hwdb must consult the live commit log per release.
2. **`drivers/hwmon/dell-smm-hwmon.c` + `drivers/platform/x86/dell-smm-hwmon.c`** — 6+ fix-class commits in the last 12 months including a `Cc: stable` regression fix from `1c1658058c99` (v6.18). The "automatic fan mode" introduction is still settling; expect more whitelist additions per release.
3. **`drivers/hwmon/nct6775-core.c`** — moderate but high-impact churn; **the most stable-tagged commits in the dataset** (rows 1, 6, 35 are LTS-relevant). The split into core + platform driver (`c3963bc0a0cf`, v5.18) is still generating fallout.
4. **`drivers/gpu/drm/amd/pm/swsmu/smu13` and `smu14`** — RDNA3 fan_curve and RDNA4 fan readback are still being filled in. Each new ASIC variant (Strix Halo, Radeon Pro variants) lands another fan-wiring commit. Expect **v7.0/v7.1 to be the first stable kernel** for full RDNA4 fan control.
5. **`drivers/platform/x86/thinkpad_acpi.c`** — slow cadence but high blast radius. The FANG/FANW mechanism added in v6.16 is still being calibrated per DMI (T495*/E560 disable in v6.17, X120e RPM scaling in v6.14).
6. **`drivers/platform/x86/alienware-wmi-wmax`** — 5 commits in late 2025 / early 2026 expanding to G15 5530, 16X Aurora, Area-51, m18, and Alienware 16 Aurora.
7. **`drivers/platform/x86/hp-wmi.c`** — Victus / Omen manual PWM (v6.20) is the active frontier; v7.1 adds locking. Expect more board adds.
8. **`drivers/hwmon/cros_ec.c`** — Framework Laptop / ChromeOS-EC PWM and target-speed support landed v6.18-v7.1. The first kernel where ventd can drop the `framework_laptop` OOT recommendation is v6.18.
9. **`drivers/hwmon/nct6683.c`** — Customer-ID adds across **every release** (B650 Steel, Z590 Taichi, BC-250, B650I Lightning, X670E Taichi, multiple MSI variants). Long tail will continue indefinitely; each new board ships ~1 release after BIOS finalises.
10. **`drivers/hwmon/macsmc-hwmon`** (Asahi) — first in-tree mfd init landed v7.0; expect rapid v7.0-rc fixes through v7.1.
