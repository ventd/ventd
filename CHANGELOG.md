## [v1.3.0] - 2026-05-28

### Added

- **The `w_pred_system` global gate is now wired to real signals (R11 PR-1); predictive control yields to the reactive curve the moment it shouldn't be trusted.** v0.5.9 shipped the confidence aggregator already consuming a `w_pred_system` boolean (false ŌåÆ every channel's `w_pred` forced to 0, bypassing the Lipschitz cap, UI state "refused") but the only call site hardwired it `true`, so the "refused" pill was unreachable and the gate was effectively absent. The new `internal/confidence/gate` evaluator composes the spec ┬¦2.5/┬¦3.6 five-term AND ŌĆö KV schema loaded, idle hard-preconditions clear (not on battery, not in a container, no disk scrub, past the ~10-minute boot/resume warm-up), the wizard outcome is control-mode, no system-wide fan mass-stall, and smart mode not disabled in Settings ŌĆö into one atomic snapshot refreshed by a daemon-lifetime goroutine every 2 s, so the expensive precondition probe (sysfs globs + `/proc` walk) never runs on the per-fan hot path. The new `internal/massstall` tracker counts concurrently-stalled fans (commanded PWM Ōēź the 30%-of-255 stiction floor, observed RPM exactly 0, Ōēź 2 channels within a 3-minute window; a single dead/parked fan and tach-less reads never count) fed by a new `controller.WithStallReporter` option that reuses the tach read the stuck-fan warning already performs (zero extra sysfs reads). A new `smart.disabled` config toggle ŌĆö with a "Disable smart mode" switch on the Settings ŌåÆ Smart mode page ŌĆö forces the gate closed for operators who want pure reactive control. `/api/v1/confidence/status` now returns a `gate` block (open/closed, the first failing reason, and each term's state), and a new `w_pred_gate` doctor detector surfaces the verdict: benign reasons (boot warm-up, smart disabled, wizard not in control mode, monitor-only) read as OK so a freshly-booted host doesn't look faulty, while a concurrent mass-stall reads as Warning. The per-layer `driftFlags` remain stubbed at `[3]bool{}` pending R11 PR-2 (the EWMA-control-chart drift detector). New rule families `RULE-GATE-*` (`internal/confidence/gate`), `RULE-MASSTALL-*` + `RULE-CTRL-STALL-REPORT-01` (`internal/massstall` + the controller feed), plus `RULE-STATE-11`, `RULE-IDLE-11`, and `RULE-DOCTOR-DETECTOR-WPREDGATE`.
- **Per-layer drift detection is now wired (R11 PR-2); a confidence layer whose model has diverged from reality is detected, its confidence decays, and the controller rides back to reactive while the layer re-learns.** v0.5.9 shipped the aggregator already consuming `driftFlags[3]` (a flagged layer's confidence decays 0.5^(t/60s) before the min-collapse) but with no producer ŌĆö the R16 detector was deferred. The new `internal/confidence/drift` detector is a residual EWMA control chart (Ross et al. 2012) per (channel, layer): it monitors each layer's prediction residual against an adaptive control limit ╬╝+L┬ĘŽā built from that layer's own healthy baseline, trips on a sustained excursion (with warmup + dwell hysteresis so it never flickers), and freezes the baseline while flagged so the anomaly can't poison its own reference. EWMA's deliberate lag on abrupt change is covered by the PR-1 `w_pred_system` gate (which drops predictive instantly); the detector's remit is gradual drift ŌĆö a re-cabled fan, dust load, ambient shift, or curve edit. Layer A's `RMSResidual` and Layer C's `EWMAResidual` were already exposed; Layer B (`internal/coupling`) now folds its RLS innovation into an `EWMAResidual` on its Snapshot (behaviour-preserving ŌĆö `Confidence()` doesn't read it). The blend hook passes the detector's flags straight into `aggregator.Tick` (never `SetDrift`, to keep the decay clock single-sourced). `/api/v1/confidence/status` gains per-channel `drift_active` + per-layer `drift_a/b/c` evidence (residual vs control limit, nil/"ŌĆö" until the layer converges); the dashboard confidence popover shows the drifting layer's real numbers, and the smart-mode page's existing "drift active" pill becomes reachable. Conservative HIL-tunable defaults (L=3, TripDwell=3, ClearDwell=5, WarmupTicks=30, a MinSigma floor): a false trip only loses predictive benefit, and a missed drift is bounded by the aggregator's clamp + Lipschitz cap. New rule family `RULE-DRIFT-*`.

### Changed

- **`--enable-gpu-write` opt-in flag removed; NVIDIA GPU writes ship enabled by default.** The flag was the last surviving HIL-style opt-in gate from the v0.6.1 sweep that removed `--unsafe-corsair-writes` and `--enable-nbfc-write`. The remaining safety mechanism is the per-device capability probe (`RULE-GPU-PR2D-01`) ŌĆö it refuses writes when the NVML symbols `nvmlDeviceSetFanSpeed_v2` / `nvmlDeviceSetFanControlPolicy` are absent (pre-Maxwell hardware or pre-R515 NVIDIA driver), which is a real driver-level constraint, not an opt-in policy decision. The laptop-dGPU detection (`RULE-GPU-PR2D-06`) and the AMD-OverDrive experimental gate (`RULE-EXPERIMENTAL-AMD-OVERDRIVE-01`) remain in place because they encode distinct constraints. Operators who previously needed to pass `--enable-gpu-write` to drive their desktop NVIDIA GPU's fan now get fan control on a fresh install with no extra step. Rule wording, registry comments, and the two now-irrelevant subtests (`write_refused_without_flag` and `write_allowed_with_flag_and_rw_quirk`) are dropped.

### Fixed

- **The installer no longer false-warns "NVML verification failed ŌåÆ GPU features may be disabled" on a working GPU.** The post-install NVML check hard-coded `sudo -u ventd nvidia-smi`, but the shipped unit runs `User=root` ŌĆö so on every host where the daemon runs as root (and therefore already has full `/dev/nvidia*` access, world-readable `0666` nodes besides), the check probed a context the daemon never uses, failed, and printed a scary "GPU features may be disabled" banner plus a `usermod -aG video ventd` "fix" that was both unnecessary (root has access) and ineffective (it can't grant access to a `root:root` device). The verify now resolves the unit's actual `User=` and probes NVML in that context (directly, as root, on the shipped unit); it only warns on a genuine failure, and shows the group-membership hint only when the daemon runs as a non-root user. New `_ventd_service_user` / `_ventd_verify_nvml` helpers, covered by `validation/install-nvidia.test.sh` (now wired into CI). Follow-up to #461 (whose `SupplementaryGroups=video` fix addressed the daemon's access but left the verification misaligned). Surfaced by the v1.3.0 factory-reset HIL.
- **`ventd --probe-modules` / `--rescan-hwmon` no longer persist a monitor-only driver when a read-only chip is already loaded.** `AutoloadModules`' fast path persisted the module owning the first visible `pwmN` file the moment one was present, skipping the `pwm_enable` controllability check that `tryModuleCandidates` applies before it persists a probed winner. On a board whose in-tree driver exposes read-only PWM (an NCT6687D bound by `nct6683`, an IT8688E under a partial `it87` bind), re-running the probe while that driver was already loaded wrote `/etc/modules-load.d/ventd.conf` pinning the monitor-only driver for boot ŌĆö masking the out-of-tree driver the wizard's `DriverPlan` would otherwise propose, and (because an explicit `modules-load.d` entry overrides a `modprobe` blacklist) resurrecting the in-tree driver at boot even after the wizard installed and blacklisted-around the OOT replacement. The fast path now gates on `controllablePWMPaths(...)`: when only read-only PWM is visible it falls through to the full probe sequence exactly as on a cold host, leaving the chip loaded for `Diagnose()` to surface the right out-of-tree driver. Surfaced by a factory-reset reinstall test on an MSI PRO Z690-A DDR4 (NCT6687D). (#1396)
- **The setup wizard now calibrates laptops whose only fan control is a non-hwmon HAL backend (msiec, thinkpad, nbfc, ipmi, legion, lenovoideapad, corsair, asahi, crosec, pwmsys).** The wizard's `ProbePhase` enumerated fans by globbing `/sys/class/hwmon/*/pwm[0-9]*` only, so a machine with no Super-I/O / hwmon PWM (an MSI Thin GF63 12UDX: `msi_wmi_platform` is tach-only, fan control is the `msiec` EC backend) reported zero controllable fans. The wizard sat on "Detect chip / probing Super-IO" and fell through to monitor-only, even though the running daemon was already driving the fan via the HAL registry. `ProbePhase` now also runs a `hal.Enumerate` pass and tags each non-hwmon channel with its backend; the tag flows through RPMDetect (which now scans every hwmon `fan*_input` to pair the cross-device tach the EC fan's RPM is reported on), Polarity (mode/level backends are monotonic, classified `normal` without the raw-sysfs prober), Calibrate (sweeps with the backend type and overlaid tach), and Apply (emits `type: <backend>`). The hwmon path is unchanged: a nil enumerator reproduces the prior behaviour, pinned by a test. Also persists the applied-marker on auto-run wizard completion (the in-memory-only flag left `Needed` true so the page re-triggered the wizard every boot), and demotes the per-tick `msiec` "shift_mode surface available" log from Info to Debug (it flooded the journal). (#1376)
- **Smart-mode channel ordering is now byte-stable across polls.** `/api/v1/smart/channels` and `/api/v1/confidence/status` were iterating the aggregator's map snapshots in Go's hash-randomised order, so the dashboard's per-fan tile grid reshuffled on every 2 s tick. Both handlers now `sort.SliceStable(... by ChannelID ...)` before returning, and the per-channel `marginal[]` list is sorted by signature label for the same reason. Pinned by `TestHandleSmartChannels_StableOrder_AcrossPolls` and `TestHandleConfidenceStatus_StableOrder_AcrossPolls` (5 consecutive polls must return identical ordering across non-alphabetic inputs).
- **Doctor `calibration_curve_quality` finding no longer misattributes the cause to "Dell SMM / ASUS Q-Fan / HP Omen firmware".** The hardcoded vendor-EC wording was wrong on every super-IO desktop board (NCT668x, IT87xx, F71xxx) and on every laptop OEM other than the three named. New wording lists candidate causes neutrally ŌĆö "firmware reasserting its own fan curve above a PWM threshold on some laptops, motherboard super-IO tach quantisation, a fan that stalls and restarts in part of the range, or a noisy tach signal" ŌĆö and the test gains a banned-substring assertion that fails the build if the named OEMs creep back into the detail text.
- **Smart-mode acoustic surface now self-describes whether `current_dba` is measured or estimated.** Pre-fix, `/api/v1/smart/status` exposed `current_dba` as if it were dBA at the mic regardless of whether a per-host R30 calibration record was loaded; consumers had to infer the meaning from the sibling `mic_calibrated` bool. New explicit `mode: "measured" | "estimated"` field on the acoustic payload. When `mic_calibrated=false`, `mode="estimated"` makes the RPM-derived nature explicit to every client without a separate inference.
- **Topbar smart-mode badge no longer flicks `learning Ōåö warming` every ~5 min.** The marginal-shard warm-start path re-pins per-channel UI state to `warming` for one or two ticks every cold-start window, and the worst-of-channels collapse propagated each transient to `/api/v1/smart/status.global_state`. New 30 s server-side hysteresis (`smartGlobalHystState`) holds a converged ŌåÆ warming/cold-start regression for up to `smartGlobalHystWindow` so the badge tracks the controller's actual state rather than the shard-reload artifact. Deteriorations to `drifting` or `refused` commit immediately ŌĆö only the known marginal-shard transient is smoothed. Three new tests pin the policy (`TestSmartGlobalHyst_HoldsBriefRegressionFromConverged`, `TestSmartGlobalHyst_DoesNotSmoothDriftingOrRefused`, `TestSmartGlobalHyst_FirstObservationCommitsImmediately`).

## [v1.2.0] - 2026-05-23

### Headline

Audit and fix wave closing 13 long-standing open issues across watchdog reliability, installer hygiene, in-UI update on non-systemd hosts, webUI viewport fit, smart-mode UX, chip-mode mismatch detection, and the first-boot polarity probe leak. 14 PRs landed (10 issue PRs in groups A through I, plus 4 hotfix PRs for CI/release plumbing surfaced during the cycle: gofmt drift, gofmt-resolution hardening, tumbleweed re-disable, flaky-test registry add). Highlights: `LastKnownStore` key is now rmmod+modprobe-stable (was keyed by the volatile `hwmonN` path), `SafePreDaemonEnable` walks `{2, 99, 0}` instead of hard-coding `2`, `hwmon.dynamic_rebind` defaults on, in-UI update surfaces failures on OpenRC/runit and works with `User=ventd`, webUI pages fit the desktop viewport with a sticky topbar, `Fan.DisplayLabel` lets operators override hwmon-reported labels, `scheduler.timezone` pins schedule semantics across DST and geographic moves, and the smart-mode page gets a preset switcher plus first-visit warming banner.

### Added

- **`scheduler.timezone` config knob (#624).** Schedule strings like `22:00-07:00 mon,tue` used to be interpreted in the daemon's Local time; a laptop / NAS that moved geographic timezones or hit a DST transition silently shifted the schedule window without any operator-visible signal. The new top-level `scheduler.timezone` config field accepts `local` (the back-compat default ŌĆö empty value resolves the same way), `utc` (recommended for new installs because it doesn't drift), or any IANA name (e.g. `Australia/Sydney`). The scheduler converts the live wall clock into the configured zone via `time.In()` before evaluating `Schedule.Matches()`. Unrecognised names log a one-time WARN at daemon startup and fall back to Local. `/api/v1/schedule/status` now includes a `timezone` field carrying the resolved label so the Schedule page can show operators which zone their schedule strings are evaluated against. Per RULE-SCHEDULE-TZ-01.
- **`Fan.DisplayLabel` override for hwmon-reported labels (#631).** The hwmon-reported `fan*_label` (CPU_FAN, CHA_FAN1, etc.) sometimes carries values that don't match the operator's case layout. `DisplayLabel` is an optional yaml/json field on each `Fan` config entry ŌĆö when non-empty it supersedes `Name` for display purposes only (Control bindings, watchdog identity, and every internal reference continue to key off the stable `Name`). The status API's new `fanStatus.Label` field carries `DisplayLabel || Name` so the dashboard fan tiles render the override transparently; the dashboard JS prefers `label` over `name` on every tick, so an operator who edits `display_label` in `config.yaml` and reloads sees the rename without a page refresh.
- **Chip-mode mismatch detection (#759 layer 1 of 2).** Adds the post-sweep flat-RPM-across-sweep detector that flags the canonical 3-pin-fan-on-PWM-mode-header signature (RPM is constant across PWM=64/128/255 because the PWM signal is being interpreted as a constant DC drive ŌĆö the fan never slows). `DriverProfile.PWMModeWritable` declares whether the resolved driver exposes a writable `pwmN_mode` sysfs attribute the future self-heal layer will flip to DC mode; populated for `it87` (false), `nct6775` (true), `adt7475` (true), `lm85` (true), `w83627ehf` (true), `asus-ec-sensors` (false), `dell-smm-hwmon` (false), `nct6683` (false), `thinkpad_acpi` (false). `ChannelCalibration.ModeMismatchSuspected` + `ModeMismatchEvidence` carry the detection through the artifact chain; `CalibrateFanResult` mirrors the fields so `ApplyPhase` can read them. `ApplyPhase` now excludes mode-mismatch-suspected channels from active control and surfaces them through the uncontrollable list with a `mode_mismatch_suspected:<evidence>` reason ŌĆö operators see the verdict in the doctor surface (#757) rather than the channel silently failing to respond. The self-heal write (pwmN_mode=0 + re-sweep on writable drivers) is the load-bearing follow-up layer; this PR ships the detection + surface so a fan no longer fails silently.
- **Smart-mode page actionability + first-impression rewrite (#1254 MVP).** The page was passive telemetry with no path from "Smart mode is learning" to "and here's what you can do". Three concrete additions: (a) **Preset switcher** ŌĆö three-position Silent/Balanced/Performance segmented switch directly under the header that PUTs `/api/v1/confidence/preset`, turning the page from observation-only into a usable control surface. (b) **First-visit warming banner** ŌĆö when `global_state=warming` AND `converged=0` the page renders a friendly amber band: *"Smart mode is learning your system's thermal behaviour. This usually takes 1ŌĆō6 hours of normal use. Come back later, or pick a preset above to adjust the balance between quiet and cool."* Dismissible per-session via `localStorage`. (c) **DisplayLabel-aware fan names** ŌĆö `fanNameFor` now resolves via `Fan.Display()` so the operator-overridable `display_label` from #631 surfaces in smart-mode strips + confidence cards. The preset switcher is wired to the same backend the legacy Settings preset toggle uses; no new endpoints. Also fixed a flake in the nohup-path watcher's sentinel reader (`update_outcome.go`): empty content during a partial-read race now retries on the next tick instead of returning early with no outcome.

### Fixed

- **First-boot polarity auto-probe no longer leaks `pwm_enable=1` across daemon exit.** In first-boot mode (no `/etc/ventd/config.yaml`, `cfg.Fans` empty) the polarity auto-probe goroutine wrote to `pwm*` paths that no controller had Registered with the watchdog, so the daemon-exit `wd.Restore` was a no-op and the channel stayed in manual mode pinned to the last probe write. Masked on dell-smm hardware (SMI-private fan control ignores sysfs `pwm_enable`); real on any direct-control hwmon backend (NCT6687, IT87, k10temp-based ECs). Fix hoists watchdog ownership from `runDaemonInternal` up to `run()` so the deferred `Restore` is in scope before any background PWM-writing goroutine spawns, and threads the watchdog into a new `runPolarityAutoProbe` helper that Registers each channel before the prober's first write and `RestoreOne`+`Deregister`s after probe completion (so subsequent controller registration captures `pwm_enable=auto` as its own orig). `polarityWG` ordering guarantees the goroutine has fully exited before the daemon-exit Restore runs, eliminating the write race between the goroutine's last probe write and the safety-net Restore. (#1312)
- **Watchdog `LastKnownStore` key now survives `rmmod`+`modprobe`.** The pre-#1331 KV key embedded the volatile `/sys/class/hwmon/hwmonN/` prefix; udev reallocates that number across module reloads (DKMS upgrade, driver-fork testing, kernel upgrade) so the persisted pre-daemon `pwm_enable` became unreachable after every rebind and the prior-crash fallback path was forced to guess. The new key shape is `watchdog.<chip>.<bus>.pwm<N>.preDaemonEnable` (e.g. `watchdog.nct6687.2592.pwm1.preDaemonEnable`) ŌĆö both components survive module reload. `ChannelIdentity` carries the legacy path alongside the new identity so the migration shim picks up pre-upgrade entries on first read and deletes them on first write. The `LastKnownStore` interface is now wired in production via `state.KVDB`; previously `NewWithStore` existed but had no production caller, so the recovery path was dead code. (#1331)
- **Watchdog `SafePreDaemonEnable` walks a fallback sequence instead of hardcoding `2`.** The kernel ABI defines `pwm*_enable >= 2` as a range, not a single value ŌĆö most in-tree drivers accept `2` but NCT6687D-pre-#169 (and the absorbed vendor-EC forks under github.com/ventd/) historically rejected `2` with EINVAL. The pre-#1332 fallback wrote `2`, hit EINVAL, dropped to manual mode (`pwm_enable=1`), and left the fan stuck at whatever PWM byte the crashed daemon had last written instead of handing back to firmware. Replaced the single constant with `SafePreDaemonEnableSequence = []int{2, 99, 0}` plus a per-entry `fallbackSeq` field on the watchdog entry that the hwmon backend's restore path walks on EINVAL. First non-EINVAL write wins and is persisted back to `LastKnownStore` via the new `OnEINVALRecovery` callback so the next prior-crash recovery skips the walk. The sequence is applied identically across all chips ŌĆö no per-chip lookup, no reordering ŌĆö so the fragility the old constant introduced is gone. (#1332)
- **`hwmon.dynamic_rebind` defaults to enabled and the swap monitor now dispatches restarts on detection.** The pre-#1265 default was off, so an `rmmod nct6687 && modprobe nct6687` (DKMS upgrade, driver-fork testing) left the controller bound to a stale `/sys/class/hwmon/hwmon10/pwmN` path while udev reallocated the chip at `hwmon9`. The daemon stayed "active" but every PWM write hit ENOENT against a vanished path. The uevent-driven `RebindTrigger` (already present) is now armed by default ŌĆö explicit `dynamic_rebind: false` is the opt-out. The periodic `RULE-HWMON-SWAP-MONITOR` re-resolver now also signals `restartCh` on detection (previously observability-only), covering hosts where the netlink path is unavailable (containers without `/sys/bus` permissions, netlink subscription failure). The pointer-with-default `*bool` shape keeps "missing field means default-on" intact while still honouring an explicit opt-out from operators who want the v0.2.x behaviour back. (#1265)
- **DKMS install no longer exits 6 with the module silently de-registered.** The driver install pipeline runs `Copy` (write `.ko` to `/lib/modules/<release>/extra/`) before `RegisterDKMS` (run `dkms add`+`dkms build`+`dkms install`). `dkms install` then aborts with exit 6 ("Module ... already installed at version 1.0.0, override by specifying --force") and the install pipeline logged WARN and swallowed the failure ŌĆö leaving DKMS with `added` but no `installed` record for the current kernel. On the next kernel upgrade the DKMS auto-rebuild hook didn't fire and ventd's controller silently dropped to monitor-only mode until the operator re-ran the wizard. Fix: pass `--force` to `dkms install` so the second write is a no-op overwrite of an identical .ko (both come from the same source + same kernel headers, validated by the reproducible-build envelope from #516). The Copy step's "module loads now" path is preserved; the DKMS registration is now load-bearing for the auto-rebuild flow. (#1267)
- **v0.8.x orchestrator wizard now re-probes after driver install, ending the "permanent monitor-only" first-install state.** Same architectural class as #1108 (which wired `Manager.afterFinalize` for the legacy `Manager.run` path) but on the v0.8.x orchestrator bridge that #1108 didn't touch. The wizard's pre-install probe persisted `wizard.initial_outcome = monitor_only` to KV (drivers weren't loaded yet); the daemon-startup probe gate then short-circuited on every restart and reused the stale outcome forever. Every smart-mode subsystem (`aggregator`, `signature`, `coupling`, `marginal`, `layer_a`, `blended`, `opportunistic`) stayed inert despite controllers actively driving PWM on the newly-installed driver's fans. Fix: extract the post-ApplyPhase-success block into `onApplyPhaseSuccess`, slot `afterFinalize` between `persistOrchestratorPolarity` and `fireReloadTrigger` so the reload reads a refreshed wizard outcome. Ordering pinned by `TestOnApplyPhaseSuccess_FiresReProberBeforeReload`. (#1268)
- **`ventd-uninstall` now removes every artifact the installer drops, not just `ventd.service` + the main binary.** Pre-fix the script left ~10 artifacts behind: `ventd-nvml-helper`, `ventd-postreboot-verify.sh`, `ventd-recover`, `ventd-wait-hwmon` (helpers in the same prefix); `ventd-recover.service` (enabled-on-install ŌĆö runs at every boot against a now-vanished binary) and `ventd-postreboot-verify.service` (sibling units); `/etc/modules-load.d/ventd.conf` (the pre-fix glob was `ventd-*.conf` with a dash; the installer writes the no-dash form); the udev rules at `/etc/udev/rules.d/90-ventd-hwmon.rules` and `/usr/lib/udev/rules.d/90-ventd-hwmon.rules`; the AppArmor profiles at `/etc/apparmor.d/ventd`, `ventd-ipmi`, `ventd.compat`; and `/var/log/ventd/` install + SELinux-build logs. The new script disables every ventd-managed unit, removes the actual `modules-load.d` filename, removes every helper binary in `$VENTD_PREFIX`, unloads AppArmor profiles via `apparmor_parser -R` before deleting the files, reloads udev, and cleans `/var/log/ventd`. The 10-class coverage is pinned by `TestUninstallScript_CoversIssue1320Artifacts`. (#1320)
- **In-UI update failures on OpenRC / runit hosts now surface in the Update modal instead of silently disappearing.** The nohup-fallback spawn path (Alpine, Void, Gentoo OpenRC, any host without `/run/systemd/system`) had no outcome watcher: operators saw "scheduled" then the daemon either restarted (silent success) or stayed on the old version forever (silent failure). The install log existed at `/var/log/ventd-update.log` but the UI never read it. Fix: the nohup wrapper now writes the install.sh exit status to `/run/ventd/update.exitcode` after the install completes; a Go-side `watchUpdateApplyOutcomeNohup` watcher polls the sentinel and captures non-zero exits into `LastApplyOutcome` along with the tail of the install log. `/api/v1/update/check` surfaces it via the existing `last_apply_error` field. Frontend updated to render a persistent "Last update attempt failed" row with a collapsible install-log tail. Symmetric with the systemd-run path's `journalctl -u ventd-update.service` tail. (#1305)
- **In-UI update no longer breaks when `ventd.service` runs as `User=ventd`.** The daemon's `systemd-run` call goes via D-Bus and `org.freedesktop.systemd1.Manager.StartTransientUnit` is polkit-gated. On root the call is auto-approved, but the hardened `User=ventd` shape failed with `auth_admin` (the daemon has no TTY / agent) and the `/api/v1/update/apply` request returned 500. Fix: ship `deploy/50-ventd-update.rules` and have `scripts/install.sh` drop it into `/usr/share/polkit-1/rules.d/`. The rule is scope-limited to the `ventd` user and the single transient unit name `ventd-update.service` ŌĆö the daemon still cannot start arbitrary system services. `scripts/uninstall.sh` removes the rule on uninstall. Latent today (current `deploy/ventd.service` ships `User=root`) but unblocks the long-tracked `User=ventd` transition. (#1306)
- **WebUI pages fit the desktop viewport without scrolling the whole page.** Pre-fix `.shell-layout` used `min-height: 100vh`; the body grew with content and the browser scrolled the whole page, so the topbar crumbs slid out of view on long fan lists or settings sections. Fix: pin `.shell-layout` to `height: 100vh`, give `.shell-main` `overflow-y: auto` so the main column is now the scroll surface, and make `.topbar` `position: sticky` inside it so the operator never loses the page header to scrolling. Mobile (<900px) keeps the previous scroll-the-page behaviour because the sidebar collapses and the extra primitive vertical real estate matters more than a pinned topbar. Added desktop-only (Ōēź1200px) padding/gap compression on `.shell-content` so more of a long list fits before scroll kicks in. (#749)
- **Smart-mode convergence fixes for the 16h NCT6687D test (#1253 partial).** Two concrete bugs from the long-running NCT6687D shake-out: (a) Layer-C marginal-benefit estimators on near-singular Fisher matrices emit conf_c values on the order of 1e-150 that propagate through the min collapse + LPF into `w_pred` at the same magnitude ŌĆö the reported `confidence_min` becomes an exponential artifact and the blender's `w_pred = 0 ŌåÆ pure reactive` branch never fires. Added `WPredUnderflowFloor = 1e-6` clamp in the aggregator that pins sub-floor `w_pred` to exact zero. ### Fixed

- **Dell Latitude 7280 (and structurally similar Dell SMM laptops) no longer end the wizard in monitor-only mode on a cold-boot run.** The polarity probe on the cold-idle dell_smm EC was returning ╬öRPM=0 because the firmware was refusing to spin the fan at idle temperatures ŌĆö ventd treated this as terminal phantom, calibrate skipped the channel, apply admitted zero fans, and the WebUI sat at "0 fans calibrated ┬Ę curve ready" with no operator explanation. With Layer 1 + Layer 2 of the probe rework, the channel is admitted, the daemon writes through, and the WebUI explicitly tells the operator what happened and why.
- **Opportunistic probing no longer refused every tick with `proc_interrupts_unreadable`.** `deploy/ventd.service` set `ProcSubset=pid` alongside `ProtectProc=invisible` since the original sandbox-hardening pass (#6, long before opportunistic probing was a feature). `ProcSubset=pid` hides everything in `/proc` except per-PID directories from the daemon's mount namespace, so `ReadIRQCounters("/proc/interrupts")` returned ENOENT, the soft idle gate refused every tick with `ReasonProcInterruptsUnreadable`, and `GET /api/v1/probe/opportunistic/status` surfaced the sentinel on the dashboard. - **Opportunistic probes spuriously aborted on routine sensor noise.** `abortOnSlope` computed raw single-sample dT/dt with no smoothing; Intel coretemp / AMD core temps / EC laptop sensors are quantized to 1 ┬░C and produce 2ŌĆō4 ┬░C single-sample steps at idle from brief turbo bursts or per-core readback latching. At 10 Hz a 4 ┬░C step is 40 ┬░C/s ŌĆö 20├Ś the laptop class 2 ┬░C/s ceiling ŌĆö so every probe aborted within ~3 samples. (#1322, #1327)
- **Per-class soft-mode PSI ceilings.** The previous global 10 % CPU PSI ceiling was correct for laptops (the user notices fan ramps so anything above 10 % avg60 is no longer "idle") but refused every probe on any 24/7 services box where steady-state services produce 20ŌĆō40 % avg60 as their normal idle baseline. Background daemon work (snapd updates, dnf metadata refresh, browser tabs) routinely pushed it above. (#1329)
- **Blocklist baseline tolerance.** The process blocklist (`internal/idle/cpuidle.go`) lists transient heavy work: rsync, restic, borg, ffmpeg, plex-transcoder, jellyfin-ffmpeg, x265, HandBrakeCLI, dnf, etc. Correct on a laptop where these are one-off jobs; wrong on a homelab where ffmpeg / plex-transcoder are 24/7 services. The soft gate now compares the current snapshot against a per-tick `ProcessBaseline` (caller-owned, scheduler-heap, same pattern as `IRQBaseline`): processes present in the previous tick are tolerated as steady-state; processes new in the current tick refuse but immediately absorb into the baseline so an hour-old rsync becomes steady-state. Strict mode (`--strict-idle-gate`) keeps the old refuse-on-any semantics. (#1330, #1333)
- **Dashboard reason strings made operator-readable.** The smart-mode page rendered raw refusal sentinels ŌĆö `recent_input_irq:irq=1`, `psi_pressure`, `proc_interrupts_unreadable` ŌĆö straight from internal constants. Detail suffixes are folded where actionable (`blocked_process:rsync` ŌåÆ "Blocking process running: rsync") and dropped where they're noise (`irq=1` ŌåÆ just "keyboard or mouse"). (#1329)
- **Input-IRQ classifier no longer matches USB host controllers.** `inputIRQClassifierKeywords` included `xhci_hcd`, `ehci_hcd`, `uhci_hcd`, `ohci_hcd` ŌĆö the USB HCI driver names. `/sys/kernel/irq/<id>/actions` lists only the IRQ handler (the HCI driver), regardless of which USB devices are attached. So matching the HCI name refused on ANY USB activity (storage scans, audio DACs, UPS heartbeats, autosuspend wake-ups). Verified on proxmox: IRQ 131 (xhci_hcd, no HID device) refused every other tick with `recent_input_irq:irq=131`. The remaining classifier entries (`i8042`, `hid`, `usbhid`, `kbd`, `mouse`, `synaptics`, `elan`) cover per-device input drivers that genuinely indicate human input. With `SlopeAbortGate` filtering noise (#1327), the slope abort only fires on real thermal events ŌĆö and probing PWM=0 on any thermally-loaded host reliably is one (fan stops, CPU warms, abort fires, correctly). RULE-OPP-PROBE-06's "aborted opp probes stay eligible for retry" was written for transient workload spikes, not for structural cooling removal, so the scheduler picked PWM=0 again every tick forever. - **Per-class `SlopeAbortConsecutive` so HEDT / server probes survive workload bursts.** The global `DefaultSlopeAbortConsecutive = 3` (0.3 s at 10 Hz) is the right number for a laptop's small thermal envelope ŌĆö any sustained rise IS dangerous. Added `SlopeAbortConsecutive` to `envelope.Thresholds`, populated per class: laptop 3, mid_desktop 4, mini_pc 4, nas_hdd 5, hedt_air / hedt_aio 6, server 8. `SlopeAbortGate.ShouldAbort` resolves the threshold via gate-override ŌåÆ per-class ŌåÆ default ŌĆö preserves all existing test wirings without modification while production gets the per-class calibration via the existing `LookupThresholds(deps.Class)` plumbing. ## [v0.5.29] - 2026-05-08

### Headline

The in-UI updater actually works. v0.5.x had a latent two-stage bug ŌĆö the staged install.sh was unreachable from the spawned transient unit, AND when that unit failed the API still replied 202 "scheduled" ŌĆö so operators clicking "Update" in the dashboard saw a green ack and watched nothing happen. ### Fixed (operator-visible)

- **`writeInstallShBytes` stages install.sh under `/run/ventd`, not `/tmp`** (#1006). ventd.service ships `PrivateTmp=yes`, so the daemon's view of `/tmp` is a per-unit kernel namespace. The transient `ventd-update.service` spawned via `systemd-run` runs in the host namespace; a script staged in the daemon's PrivateTmp `/tmp` is not at that path on the host, and bash returns `exit 127` / ENOENT. systemd journal records `ventd-update.service: Main process exited, code=exited, status=127/n/a` but the API caller saw a successful 202 because `realUpdateRun`'s `cmd.Run()` observed a successful systemd-run *queue*, not the unit's runtime exit. Diagnosed end-to-end on the MSI Z690-A desktop on 2026-05-08; latent since the systemd-run pattern landed. Fix stages under `/run/ventd` (host-shared, not namespaced under PrivateTmp, already in the unit's `ReadWritePaths`, ephemeral so no orphan litter). Falls back to `os.CreateTemp("", ...)` when `/run/ventd` is unavailable (dev-tree invocation, non-systemd hosts) so existing dev workflows keep working. Bound to new `RULE-WEB-UPDATE-STAGE-PATH-OUTSIDE-PRIVATETMP` with four subtests.

- **Failed transient unit surfaces via `/api/v1/update/check.last_apply_error`** (#1007). Even with the staging path fixed, `POST /api/v1/update/apply` previously replied 202 regardless of whether the spawned unit would actually succeed at startup. New bounded watcher goroutine (`watchUpdateApplyOutcome` in `internal/web/update_outcome.go`) polls the transient unit for up to 60 s at 1 s intervals; on `Result != "success"` and `SubState Ōłł {exited, dead, failed}`, captures the result + last 30 journal lines into a package-level `atomic.Pointer[LastApplyOutcome]`. The next `GET /update/check` includes `last_apply_error: {at, version, status, detail, journal_tail}` so the operator sees the actionable failure cause without leaving the dashboard. Success is silent (the daemon's restart is the success surface; recording success would persist a stale "last failed" indicator after the next successful install). Timeout is silent (operator can re-poll). The `omitempty` tag preserves backward compatibility ŌĆö older UIs that don't know about the field see no behaviour change. Bound to new `RULE-WEB-UPDATE-STATUS-FIDELITY` with six subtests.

### Internals

- New package-level seams in `internal/web/update.go` and `internal/web/update_outcome.go`: `installStagingDir`, `systemctlIsFailedFn`, `journalctlTailFn`, `updateOutcomeWatchTimeout`, `updateOutcomePollInterval`. All overridable from tests so the test suite is hermetic and doesn't sleep for real.

## [v0.5.28] - 2026-05-08

### Headline

A senior-review pass on the v0.5.26 codebase surfaced five real safety/correctness gaps, all closed in this release. Headline change: the README's "every exit path restores firmware control within two seconds" promise is now upheld by ventd's own restore path, not just by systemd's `WatchdogSec=2s` SIGKILL + `OnFailure=ventd-recover` belt-and-braces. Plus the Go toolchain catches up to 1.25.10 to clear two stdlib CVEs that landed in the vuln database mid-cycle and started failing govulncheck on every PR.

### Fixed (safety)

- **Watchdog Restore is now budget-bounded and parallelised** (#1002). `Restore()` previously walked entries sequentially with no per-write deadline; a hung sysfs ioctl on one fan blocked every subsequent fan's restore, while the heartbeat goroutine kept pinging systemd happily ŌĆö defeating the SIGKILL backstop. New `RestoreCtx(ctx)` launches one goroutine per registered entry via the swappable `restoreOneImpl` seam, selects on WaitGroup-drain vs `ctx.Done()`. On deadline-exceeded a `restoreGracePeriod` (100 ms) lets in-flight goroutines finish their log emit, then snapshots the abandoned set, sorts deterministically, emits one WARN naming the abandoned channels + cancellation cause, and returns. The legacy `Restore()` now wraps `RestoreCtx(context.WithTimeout(_, DefaultRestoreBudget))` (1.8 s ŌĆö 200 ms headroom under typical systemd `TimeoutStopSec=2s`); existing call sites in `cmd/ventd/main.go`, `cmd/ventd/runsetup.go`, and `internal/controller/controller.go` pick up the budget without per-caller code changes. Per-entry panic recovery (`RULE-WD-RESTORE-PANIC`) and every-entry-touched (`RULE-WD-RESTORE-EXIT`) contracts hold verbatim. Bound to new `RULE-WD-RESTORE-BUDGET`.

- **NVML init bounded at 2 s to prevent startup hang on partial driver installs** (#1000). A wedged `purego.Dlopen` on `libnvidia-ml.so.1` ŌĆö caused by mismatched DKMS, stale `.so` symbols, or a hung kernel module ŌĆö would block daemon startup indefinitely past systemd's `TimeoutStartSec`, leaving the operator with "ventd failed to start" and no actionable diagnostic. New `nvidia.InitWithDeadline(ctx, logger, timeout)` wraps `Init` in a goroutine with a select on done-channel, `time.After(timeout)`, and `ctx.Done()`. On timeout fire: returns wrapped `ErrLibraryUnavailable` with `"timed out"` in the message; emits one WARN. The orphan dlopen goroutine continues running because `purego.Dlopen` is uncancellable ŌĆö once a timeout fires, NVML is permanently disabled for this process lifetime, by design. `cmd/ventd` and `cmd/ventd-nvml-helper` use the deadline-bounded variant with a 2 s budget; `nonvidia` build tag gets a matching stub. Bound to new `RULE-GPU-PR2D-09`.

### Fixed (correctness)

- **KV writes refuse before mutating in-memory state when the state directory is critically low on disk** (#1003). Prior behaviour: `KVDB.Set` mutated `db.data[ns][key] = value` first (`kv.go:100`) and called `persist()` second (`kv.go:101`). On a full `/var/lib/ventd`, `persist` returned ENOSPC to the caller but the in-memory map was already advanced. `Get` returned the new value while a daemon restart loaded the OLD on-disk value back ŌĆö silent in-memory/on-disk divergence for `wizard.initial_outcome`, calibration state, polarity records, and the smart-mode shard root. New `iox.EnsureFreeSpace(path, minBytes)` is a `statfs(2)`-based pre-flight gate; `iox.MinFreeBytesForState` (1 MiB) is the canonical state-class threshold. Plumbed into `KVDB.Set` / `KVDB.Delete` / `KVDB.WithTransaction` via the `ensureFreeSpaceFn` package-level seam (production points at `iox.EnsureFreeSpace`; tests inject a refusing stub). The gate fires before the mutex acquire and before any in-memory mutation, so refusal never advances the in-memory state ahead of the on-disk file. `WithTransaction`'s gate fires before the caller's `fn` closure even runs. Bound to new `RULE-IOX-02` and `RULE-STATE-12`.

### Fixed (supply chain)

- **Go toolchain bumped 1.25.9 ŌåÆ 1.25.10 to clear two stdlib CVEs** (#1001). govulncheck on `pre-release-check.yml` started failing on every PR after the vuln database picked up two Go 1.25.9 fixes:

  Both fixed in `net@go1.25.10` and `net/http@go1.25.10`. ### CI / hygiene

- **`.golangci.yml` pins golangci-lint v2.1.6's standard linter set explicitly** (#1004). CI's lint job (`ci.yml:347`) installs golangci-lint v2.1.6 from source and runs `golangci-lint run --timeout=5m` against the implicit default linter set: `errcheck`, `govet`, `ineffassign`, `staticcheck`, `unused`. Root cause this release closes: **`ventd.service` ships with `PrivateTmp=yes` + `ProtectSystem=strict` + a `ReadWritePaths=` list that excludes `/usr/local/bin` and `/var/log`**. The naive `nohup`-fork in `realUpdateRun` put `install.sh` inside the daemon's cgroup and inherited the entire sandbox. install.sh failed the moment it tried to write the staged binary at `/usr/local/bin/.ventd.new`, redirect to `/var/log/ventd-update.log`, or run `dpkg -i`. ### Fix
- Spawn install.sh via `systemd-run --no-block --collect --unit=ventd-update --service-type=oneshot --property=KillMode=process --setenv=VENTD_VERSION=ŌĆ” --setenv=VENTD_SKIP_PREFLIGHT_CHECKS=ŌĆ” bash <script>`. This creates a transient SERVICE unit **outside** ventd.service's cgroup with its own clean root sandbox view. `KillMode=process` keeps install.sh alive when it triggers `systemctl try-restart ventd`, which would otherwise SIGTERM the entire spawning cgroup. `--collect` frees the unit on completion so successive update attempts don't accumulate failed transient units.
- Non-systemd hosts (Alpine OpenRC, Void runit) fall back to the original `nohup` path ŌĆö those hosts don't impose a service-unit sandbox to begin with.

### Tests
- `TestBuildUpdateCmd_PrefersSystemdRun` pins the load-bearing flag set when systemd is the running init.
- `TestBuildUpdateCmd_FallsBackToNohup` covers all three "systemd not usable" edges (init not running, binary missing, both).

### Why this wasn't caught before
The chicken-and-egg work in #950/#951/#955/#960 fixed install.sh's *contents* ŌĆö assuming install.sh would run. The test harness invokes install.sh directly, never via the daemon's `exec.Cmd`.
