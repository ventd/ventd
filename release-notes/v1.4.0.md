## v1.4.0 — Hardware safety backstops, first-boot security, and a Doctor that acts

This is a big one. The headline work is a set of **curve-independent safety backstops**
(fans forced to full speed near critical temperature, no matter what any curve or mode
says), a **hardened first-boot security model**, **AMD GPU fan control wired end-to-end**
(including RDNA3/4 curve programming), and a Doctor page that now **fixes problems
instead of just describing them**.

Validated end-to-end on real hardware (13900K + RTX 4090, NCT6687D) plus a new
synthetic-hardware harness that lets the full daemon run against simulated fans.

### Safety backstops

- **Over-temperature failsafe (#1442).** If a control sensor crosses its critical
  threshold (hwmon crit, or CPU Tjmax where the chip exposes none), the fan is forced to
  true full speed — bypassing the curve, smart mode, the schedule, and even your
  `max_pwm` noise cap. Debounced and hysteresis-separated so a thermally-limited CPU that
  *lives* at Tjmax never false-fires. Covers mix-curve fans too: a `max(cpu, gpu)` case
  fan goes to full speed if *either* component crits.
- **A disconnected temperature sensor can no longer trick a fan into under-cooling
  (#1449),** and the systemd watchdog now genuinely catches a stalled control loop
  instead of pinging through it (#1446).
- **Fans follow a runtime hwmon renumber (#1497).** An `rmmod`/`modprobe`, DKMS upgrade,
  or USB hotplug that shuffles `/sys/class/hwmon` no longer quietly strands a fan on the
  BIOS curve until restart — the controller cohort drains, rebases, and respawns.
- **Re-setup no longer strands fans in manual mode (#1456),** and `ventd-recover` hands
  fans back to firmware auto on crash instead of pinning them (#1434).

### Security

- **First-boot enrolment from the network now requires a one-time setup token.** Until
  an admin password is set, whoever reaches the listener first owns the daemon — that
  window is now closed for LAN clients. The token is logged to the journal, written to
  `/run/ventd/setup-token`, and printed by the installer; enrolment from the host itself
  (localhost / SSH tunnel) stays tokenless. Retired the instant a password is set.
- **`install.sh` now cosign-verifies releases** (keyless, pinned to the CI identity),
  not just checksums. `VENTD_REQUIRE_SIGNATURE=1` makes it fail-closed.
- **Diagnostic bundles no longer include the admin password hash**, and the web server
  pins a TLS 1.2 floor.

### Shadow mode (#1346)

Evaluate ventd against your existing fan controller without ceding a single PWM:
`apply.shadow: true` runs the full pipeline — learning, confidence, decisions — but
issues **zero hardware writes**. The decisions feed shows exactly what ventd *would*
have done, so you can run it next to CoolerControl / fan2go / thinkfan for days before
flipping the switch.

### GPUs and laptops

- **AMD GPU fan control is wired end-to-end (#1454).** RDNA1/2 cards get per-tick PWM
  control (with `--enable-amd-overdrive`); **RDNA3/4 (RX 7000/9000)** get their hardware
  fan curve programmed through `gpu_od/fan_ctrl` — previously monitor-only. Not yet
  HIL-validated on real AMD silicon; reports welcome.
- **ASUS notebooks**: a writable `asus-wmi` custom-fan-curve backend (kernel 6.4+) with
  g-helper's silent/balanced/turbo presets vendored as references. Same caveat: fake-sysfs
  tested, HIL wanted.
- **Framework laptops**: `cros_ec_hwmon` catalog row (writable fan duty on kernel 6.18+)
  plus the fw-fanctrl curve presets, surfaced in `ventd doctor`.
- **3-pin/DC fans stuck on a PWM-mode header self-heal (#759)** on chips that allow it:
  flip to DC mode, re-sweep, verify, persist — or get told exactly which BIOS menu to fix.

### Doctor & observability

- **"Apply fix" buttons (#1474):** findings now carry their real remediation — install
  kernel headers, reload AppArmor, the GRUB `acpi_enforce_resources=lax` fix — plus a
  "Learn more" wiki link. No invented handlers; classes without a backed action get the
  diagnostic-bundle fallback only.
- **New detectors:** a BIOS fighting ventd for fan control (EBUSY storms), a temperature
  sensor frozen at a believable value while the box heats up, AppArmor profile drift,
  kernel-changed-under-ventd, and whether your board is in the hardware catalog.
- **`ventd health`:** a one-line "why won't it start?" probe that reads the recorded
  startup fatal when the daemon (and so the web UI) is down.
- **Prometheus `/metrics` (#1463):** fan PWM/RPM, temperatures, build info — scrape it
  straight into Grafana, no exporter needed.

### Web UI fixes

- **Settings → Calibration renders instead of a blank pane (#1414)** — the second half of
  the v1.3.0 navigation fix; the section now actually displays when selected.
- **The Settings diagnostic-bundle button generates and downloads the bundle (#1502)**
  instead of telling you to go run a CLI command.
- **`ventd setup --reset` now exists.** The login screen has always advertised it for a
  forgotten password; the command was never implemented. It clears the admin password
  (fan config and calibration are kept) and re-opens the wizard's password step — with a
  fresh one-time setup token for non-localhost access.
- The calibration "signal scope" plots the real sweep, not a canned waveform (#1469); the
  LAN URL survives a reboot (#1465); monitor-only dashboards guide you into setup instead
  of dead-ending (#1472); smart mode no longer silently targets 70 °C when no setpoint is
  configured, and its below-curve relax is bounded (#1459).

### Upgrade notes

Fully backward-compatible — no config changes required. New optional keys:
`apply.shadow`, `smart.max_relax_below_curve`. The setup token only affects *first-boot*
enrolment on fresh installs; existing installs with a password set never see it.
`ventd-uninstall` now removes only the kernel modules ventd itself installed (#1436).
