# ventd v0.2.0

ventd v0.2.0 is the first release to ship with the web UI feeling like
a real application instead of a staging area. The dashboard streams fan
state live, the typography and icons are consistent across every
browser, and the daemon now survives the kind of operator mistakes and
motherboard quirks that broke v0.1.x — rigs that expose two identical
chipsets, installs where someone ran the daemon by hand as root, and
setup runs that hit edge cases the wizard wasn't checking for.

## Highlights

- **Live dashboard.** Fan PWM, RPM, and temperature updates now stream
  over Server-Sent Events instead of being polled. The dashboard
  reflects what the daemon sees in near real time.
- **Modern UI pass.** Body text renders in the system sans-serif stack
  with monospace reserved for numeric readouts; placeholder Unicode
  glyphs replaced with a bundled Lucide SVG icon sprite. No webfonts,
  no CDN fetches, consistent rendering everywhere.
- **Install now survives sudo.** The daemon's config writer keeps
  `/etc/ventd/config.yaml` owned by the service user even if you
  invoke `ventd` directly as root. The previous behaviour silently
  locked the service out of its own config on the next start.
- **Broader motherboard compatibility.** Boards that expose two hwmon
  chips with the same driver name (e.g. dual-`nct6687` on MSI MAG Z790)
  can now start cleanly — the config file's stable device path is used
  as a tiebreaker.
- **Safer setup.** The wizard validates the configuration it's about to
  write *before* the Apply button, and no longer generates GPU fan
  curves on rigs where the GPU fan isn't actually controllable. The
  resolver pass now runs on the wizard's output as well as on daemon
  startup, so a config the daemon would reject at next boot is caught
  at Apply time instead of after a restart.
- **Responsive UI.** Layout adapts to narrow viewports: the sidebar
  collapses into a drawer, the header reflows, and breakpoint-aware
  components keep the dashboard usable on phone- and tablet-sized
  browsers.
- **Crash recovery wires correctly.** On `SIGKILL`, OOM kill, or
  panic-escape, systemd's `OnFailure=` triggers `ventd-recover.service`
  which restores `pwm_enable=1` on every hwmon channel before the daemon
  restarts. The v0.1.x `ventd.service` had the `OnFailure=` directive
  under `[Service]`, where systemd silently ignored it — so the
  "fans never stuck at a stale PWM" promise didn't actually hold for
  non-graceful exits. v0.2.0 moves the directive to `[Unit]`, adds a
  CI regression guard (`scripts/check-unit-onfailure.sh`), and has
  been verified end-to-end on the phoenix-MS-7D25 rig with live
  SIGKILL → recover cycles.

## Upgrade notes

- Re-running the installer is safe and recommended:

  ```
  curl -sSL https://raw.githubusercontent.com/ventd/ventd/main/scripts/install.sh | sudo bash
  ```

  The installer is idempotent and now refreshes the systemd unit
  files on every run (`ventd.service` and `ventd-recover.service`),
  runs `systemctl daemon-reload`, and `try-restart`s the service.
  Upgrades from v0.1.x that previously required manually re-copying
  the unit files by hand are a one-liner now. Existing configs under
  `/etc/ventd/` are preserved.

- If your existing `/etc/ventd/config.yaml` was written before the
  TLS rework in v0.1.1 (no `web.tls_cert` / `web.tls_key` keys), the
  daemon now auto-migrates on the first load after upgrade: when
  `/etc/ventd/tls.crt` and `/etc/ventd/tls.key` exist on disk and
  the config fields are empty, ventd populates them and writes the
  config back so the migration runs once. No manual edit required.
  You'll see an INFO log line the first time this fires.

- If you ran `sudo ventd …` against v0.1.x outside of the systemd
  unit at any point, `/etc/ventd/config.yaml` may be owned by
  `root:root` and unreadable by the `ventd` service user on the
  next restart. After upgrading, fix it once with:

  ```
  sudo chown -R ventd:ventd /etc/ventd
  sudo systemctl restart ventd
  ```

  New installs and normal UI-driven config saves are unaffected.

- NVIDIA users: GPU temperature reads have always worked out of the
  box. GPU fan *writes* still require the one-time udev rule (or one
  of the alternatives) described in
  [`docs/nvidia-fan-control.md`](../docs/nvidia-fan-control.md). Not
  a new limitation — now documented.

## Known issues

- Setup wizard does not yet offer to load a missing kernel module for
  you. If a sensor you want depends on a module (e.g. `coretemp`)
  that isn't in `lsmod` at scan time, the wizard will refuse to apply
  the config instead of silently writing one the daemon would reject
  at next boot ([#36](https://github.com/ventd/ventd/issues/36), fixed
  in v0.2.0). The friendlier "load this module and re-scan" remediation
  card is planned for a follow-up.

## Install

Fresh installs, one command:

```
curl -sSL https://raw.githubusercontent.com/ventd/ventd/main/scripts/install.sh | sudo bash
```

The installer detects your architecture and init system (systemd,
OpenRC, or runit), drops the binary at `/usr/local/bin/ventd`,
installs the service file, enables it, and starts the daemon. It
prints one thing at the end: the URL to open in your browser.

On first run the daemon writes a one-time setup token to
`/run/ventd/setup-token` (0600, root-only) and the controlling TTY
if one is attached. Recover it with:

```
sudo cat /run/ventd/setup-token
```
