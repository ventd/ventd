## v0.8.8 â€” ventd-org driver forks wired into the install pathway

This release switches every out-of-tree kernel-driver fetch performed by
ventd's installer over to the **ventd-org downstream forks** under
[`github.com/ventd/`](https://github.com/ventd), each pinned to a specific
release tag. The install pipeline now uses a `Tag`-based tarball URL,
so a given ventd binary version is reproducibly bound to one driver-source
revision per chip family â€” no more "tip of master" drift between two
ventd installs of the same version.

Three additional out-of-tree drivers gain reachable install paths in the
same change: ITE IT5570 EC (mini-PCs), System76 ACPI (older-kernel
laptops), and System76 Io (Thelio desktops).

Each fork carries the upstream tree on `main`/`master`, the ventd-specific
glue on a `ventd` branch (typically just a `VENTD.md` plus, for `nct6687d`,
the `blacklist-nct6683-by-ventd.conf` modprobe drop-in and DKMS install/
remove hooks), and tagged releases the installer pins to. The pin lives
in the new `DriverNeed.Tag` field; when set, the install pipeline fetches
`/archive/refs/tags/<Tag>.tar.gz` instead of the `Branch` head tarball.

### New: IT5570 / System76 install paths

`it5570_fan`, `system76_acpi`, and `system76_io` are reachable for the
first time:

- **`it5570_fan`** ([`passiveEndeavour/it5570-fan`](https://github.com/passiveEndeavour/it5570-fan))
  covers AMD Phoenix / Hawk-Point mini-PCs from Beelink, MinisForum, and
  AceMagic that previously had **no Linux fan control at all**. Detection
  is via DMI `sys_vendor` / `board_vendor` containing the brand string;
  fan stays at BIOS-auto on module unload, lining up with ventd's
  restore-on-stop guarantees.
- **`system76_acpi`** ([`pop-os/system76-acpi-dkms`](https://github.com/pop-os/system76-acpi-dkms))
  covers System76 laptops on kernels older than ~5.15 where the in-tree
  `system76_acpi` driver hasn't landed. DMI-gated on `sys_vendor` /
  `board_vendor` containing `system76`. On hosts where `system76-power` is
  already running, ventd's existing step-aside logic still kicks in â€”
  this entry just gives ventd a reliable hwmon source on older kernels.
- **`system76_io`** ([`pop-os/system76-io-dkms`](https://github.com/pop-os/system76-io-dkms))
  covers System76 Thelio desktops. DMI-gated on `system76` vendor plus
  `thelio` product so the entry doesn't surface on the System76 laptop
  range (which uses `system76_acpi`).

`Diagnose()` now reads the full DMI snapshot (`sys_vendor` and
`product_name` in addition to `board_vendor` + `board_name`) and merges
DMI-only proposals into `HwmonDiagnostics.DriverNeeds`. The new entries
ride that path; existing chip-name and board-name detection for the
NCT6687D / IT8688E / MSI EC families is unchanged.

### Reproducibility: `DriverNeed.Tag`

`internal/hwmon/autoload.go` adds a `Tag` field to `DriverNeed`. When
set, the install pipeline (`internal/hwmon/install.go`) fetches
`<RepoURL>/archive/refs/tags/<Tag>.tar.gz` instead of the `Branch` head.
Branch head is preserved as a fallback for any future entry that
intentionally tracks a moving ref. The installer's `dkms` registration
path is unaffected â€” the tag-fetched tarball still ships its
`dkms.conf`, and the existing placeholder-version detection (#785 /
#1154) still kicks in if the upstream `dkms.conf` carries an
unsubstituted `PACKAGE_VERSION`.
