# ventd v0.3.0 — plan

Status: planning. No tag date. This file is the scope anchor every
Cowork and Claude Code session reads before picking up v0.3 work; keep
it current as the cycle moves.

## Theme

Two streams, run in parallel:

1. **Mobile and cross-device polish.** v0.2.0 shipped the responsive
   pass and the 44×44 touch-target fix (#102), but the real-phone smoke
   (#68) is still outstanding and the UI was tuned primarily on the
   rig's desktop browser. v0.3 makes the web UI feel native on phones
   and tablets — not a shrunk-down desktop page.
2. **Hwmon topology resilience.** v0.2.0 closed the cold-boot udev race
   (#103) and the wrong-chip bind (#86), but both fixes are reactive.
   Proposal 3 (#98) and dynamic rebind (#95) turn topology handling
   into a first-class runtime concern rather than a startup-time
   resolve-and-pray.

Cross-distro coverage (#90) runs under both streams — every new mobile
fix and every hwmon fix is verified on at least Ubuntu, Fedora, and
Arch before it's considered done.

## In scope

### Mobile / UI

- #68 — real-phone smoke pass across Phase 4.5 viewports. Phoenix-owned
  (needs physical devices); this plan gates v0.3 on it.
- Touch-driven fan-curve editing. The curve editor currently assumes a
  cursor with hover affordances; on touch it's usable but not pleasant.
  Replace hover-only affordances with tap-and-hold, and make drag
  handles large enough to hit without zooming.
- Narrow-viewport dashboard density. The SSE-streamed dashboard shows
  every fan and sensor in one flat list; on phones this scrolls
  forever. Group by category with collapsible sections, remember state
  in session storage. **Shipped** via the `feat/mobile-dashboard-density`
  PR — Sensors and Controls render as native `<details>` accordions
  grouped by CPU / GPU / System / Storage / Other; narrow viewports
  open CPU by default with the rest tucked away, desktop keeps every
  group expanded; toggle state persists per-tab in sessionStorage
  across SSE re-renders.
- Setup wizard on mobile. The wizard was built for a desktop browser
  running next to the rig; most users will hit it from their phone on
  the same LAN. Verify every step's form layout and primary-action
  placement on narrow viewports.

### Hwmon / daemon

- #98 — Proposal 3 design → implementation. Subscribe to
  `hwmon topology change action=added` (and `removed`) and rebind the
  affected fans without restarting the daemon. Ship behind a config
  flag first (`hwmon.dynamic_rebind: true`) so it can be disabled if
  it destabilises. **Landed in #112** as an unconditional re-exec on
  `action=added`; **gated in #125** via `hwmon.dynamic_rebind` (default
  `false`), so v0.2.x behaviour is preserved until an operator opts
  in. `action=removed` remains out of scope per #112's header
  (fail-safe restore on a disappearing chip is a separate tracker).
- #95 — re-resolve on topology change. Shares code with #98; land the
  resolver changes first, then wire the subscriber. **Landed in #112**
  (#95's Option A path) and gated in #125.
- Belt-and-braces on the udev-settle replacement. `systemd-udev-settle`
  is deprecated on current distros; replace the `After=` ordering in
  `deploy/ventd.service` with a positive readiness check (the
  `ventd-wait-hwmon` helper from PR #105 becomes the canonical
  mechanism; the `After=` line goes away). **Landed in #125** —
  `After=systemd-udev-settle.service` dropped, `scripts/check-unit-
  ordering.sh` flipped to assert the new invariant (no `udev-settle`
  under `[Unit]`, `ventd-wait-hwmon` under `[Service].ExecStartPre`).
- Hwmon rescan telemetry. The 10s periodic rescan from v0.2.0 is
  silent when it does useful work. Expose the rescan's before/after
  state in `/api/debug/hwmon` so diagnosing topology drift doesn't
  require reading the journal. **Follow-up PR** — split out of #125's
  scope so the config gate and unit-file cleanup could land
  independently.

### Cross-distro

- #90 — v0.2.0 cross-distro roll-up finished in v0.2.x if it completes
  before v0.3 branches; otherwise v0.3 inherits it as a gate.
- CI matrix expansion. Build and smoke-test on Ubuntu 24.04, Fedora 41,
  Arch, and Alpine in GitHub Actions. Every PR runs the matrix; a
  single-distro green is no longer sufficient for merge. **Landed in
  #114** — `build-and-test` splits into four rows and all four are
  required on `main` by repo ruleset. Alpine runs with `CGO_ENABLED=0`
  and skips `-race` to preserve the libc-only binary guarantee from
  `.claude/rules/usability.md`; race coverage comes from the other
  three rows.

### Safety tests

- `internal/controller` safety-invariant test suite. Coverage was
  12.0 % before this work against the seven invariants in
  `.claude/rules/hwmon-safety.md` (PWM clamp, pump floor, allow_stop
  gate, pwm_enable=1, ENOENT/EIO skip, watchdog restore,
  resolve-by-hwmon_device). Each rule is now bound 1:1 to a named
  subtest so the rules are enforced by CI, not just prose. **Landed
  in #118** (tests only, two skips for known gaps), followed by
  **#124** which flipped both skips by fixing #115 (allow_stop gate
  on `config.Fan.AllowStop`) and #116 (watchdog restore on every
  `Run` exit path). Controller coverage 12.0 % → 88.0 %.

### Packaging (competitive parity, additive only)

Context: CoolerControl, fan2go, and nbfc-linux all ship AUR packages;
ventd does not. Arch users install via `install.sh` today. These
items add *distribution channels* — they do not touch `install.sh`,
`.goreleaser.yml`, the Go source, the config schema, or the web UI,
and therefore do not violate "No install-path rework" below.

- AUR PKGBUILDs (`ventd-bin` + `ventd` source) under `packaging/aur/`.
  Queued as `packaging/aur-pkgbuilds`. Publish to aur.archlinux.org
  is Phoenix-only (AUR SSH key).
- NixOS flake + module (`packaging/nix/`). Follow-up PR after AUR.
- Docker image + device-passthrough docs (`packaging/docker/`).
  Follow-up after NixOS; `/sys/class/hwmon` access from a container
  is its own design question — budget a spike.
- AppImage and Flatpak — evaluate after the above three; may defer
  to v0.4 if ROI is marginal once AUR + NixOS + Docker land.

## Out of scope

- No new hardware backends (AMDGPU-separate-from-NVML fan control,
  fan hubs, USB AIOs / liquidctl-style devices, storage-drive temps)
  — v0.3 hardens what's there, it doesn't widen it. Tracked in
  `release-notes/v0.4.0-plan.md`.
- No config-schema changes beyond the `hwmon.dynamic_rebind` flag.
  Existing v0.2.x configs load unchanged on v0.3.
- No auth or TLS work beyond bugfixes. The web UI stays HTTP-on-LAN
  with the existing one-time setup token; HTTPS remains "front with
  Nginx or Caddy". A proper auth/TLS rethink is v0.4 material.
- No install-path rework. `install.sh` + the `.deb`/`.rpm` artifacts
  from v0.2.0 carry forward unchanged. Adding *new* distribution
  channels (AUR, NixOS, Docker) is additive and is in scope under
  "Packaging" above.
- No alerts / notifications subsystem. CoolerControl parity item
  that needs a config-schema decision; v0.4 material.
