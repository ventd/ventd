# ventd v0.4.0 — plan (draft)

Status: **draft**. v0.3.0 has not branched yet. This file captures
work items that have been ruled out of v0.3 scope but are worth
keeping in one place so v0.4 planning does not start from zero. Every
entry here is re-litigated when v0.3 tags — promotion to "in scope"
requires a passing argument against the scope line at the top of the
v0.4 theme, not inclusion by default.

Source of truth for v0.3 remains `release-notes/v0.3.0-plan.md`.

## Theme (tentative)

Two candidate streams, one scope-defining choice to make at v0.3 tag:

1. **Hardware breadth.** The single biggest competitive gap vs
   CoolerControl today. Currently ventd controls hwmon PWM channels
   plus NVIDIA GPUs via NVML. Users with Corsair / NZXT AIOs, fan
   hubs, or storage-drive thermal sensors cannot use ventd. Adding a
   liquidctl-class backend and a drive-temp sensor source closes the
   gap but introduces a plugin-shaped abstraction the codebase does
   not have today.
2. **Auth / TLS rethink.** v0.2.x shipped one-time setup token +
   single admin password + optional TLS cert/key. "Front with
   Nginx/Caddy" for LAN HTTPS is an acceptable v0.2/v0.3 answer; it
   is not an acceptable v0.4 answer if ventd wants to be the
   default-install on multi-user homelabs. Per-user accounts,
   first-class ACME/Let's Encrypt, and an audit log are the three
   items that would move the answer.

Pick one stream as v0.4's primary theme and defer the other to
v0.5. Running both in parallel splits the reviewer. Leaning #1 —
hardware breadth is the gap that costs users at install time;
auth/TLS is the gap that costs users at scale, which ventd does not
yet have.

## In scope (if v0.4 = hardware breadth)

### Backend abstraction

- Introduce a `Backend` interface in `internal/controller` or a new
  `internal/backend/` package, with hwmon and NVIDIA as the first
  two implementations. No behaviour change for existing users.
- Bench against a CoolerControl-style plugin model. Rejected
  alternatives: hard-coded per-device branches (current state —
  doesn't scale), external shelled-out CLI (loses type safety,
  latency, cancellation semantics).

### USB AIO / liquidctl-class support

- Third backend: USB HID via `github.com/karalabe/hid` or equivalent
  pure-Go library. Start with one device family that covers the
  widest-deployed hardware — Corsair Commander Core / Pro is the
  most impactful first target; NZXT Kraken is second.
- Device discovery in the setup wizard: "We detected a Corsair
  Commander Core. Add its fan/pump channels?" — opt-in, never
  silent.
- Calibration semantics differ for pumps — the pump-minimum and
  allow_stop rules in `hwmon-safety.md` already cover the invariant,
  but the calibration *path* (RPM vs PWM, curve shape) needs its
  own per-device profile. Schedule a spike before committing.

### Storage-drive temperature source

- Fourth backend: NVMe temp via `ioctl NVME_IOCTL_ADMIN_CMD` (log
  page 0x02) or `/sys/class/nvme/nvmeN/hwmon*/temp*_input` if the
  `nvme` hwmon binding exists on the target kernel.
- Prefer sysfs; fall back to ioctl only when sysfs absent.
- SATA drives via SMART log (requires `smartctl` at runtime — decide
  if an optional-dep smartmontools is acceptable, or ship a native
  SMART reader).

### Config schema changes

- Backends become a typed discriminator in config:
  `type: hwmon | nvidia | usb_aio | nvme | sata`.
- Migration: v0.3 configs load as an implicit `type: hwmon` or
  `type: nvidia`; the loader infers type from existing fields and
  writes the explicit tag on next save. Users never edit YAML.

## In scope (if v0.4 = auth / TLS rethink)

### Per-user accounts

- Drop the single-admin-password model. Add users table (sqlite or
  flat JSON + fsync — prefer flat JSON for zero-deps parity).
- Role split: `admin` (everything), `operator` (start/stop,
  calibrate, edit curves), `viewer` (read-only dashboard).
- Existing single admin password migrates to a single `admin` user
  on first load after upgrade.

### ACME / Let's Encrypt

- Built-in `autocert.Manager` for users with a public DNS name
  pointing at the host. Keep "front with Nginx/Caddy" as the
  documented path for anyone behind NAT.
- Certificate material lives in `/etc/ventd/acme/` with the same
  `User=ventd` ownership model PR #6.5 established.

### Audit log

- Append-only JSONL at `/var/log/ventd/audit.log`, rotated by
  journald or logrotate.
- One event per config write, per calibration start/abort, per user
  login / auth failure, per `start/stop/restart` from the UI.

### HSTS bump

- Raise `Strict-Transport-Security` `max-age` from 300s to 1 year
  (31536000). Noted as a TODO in `internal/web/security.go:51`
  since v0.2.0; gated on "deployments stable" which v0.4 makes
  concrete.

## Alerts / notifications (both streams)

Independent of the theme choice. CoolerControl has them; ventd does
not.

- Threshold-crossing alerts (fan stuck, temp exceeds N, PWM at
  ceiling for M minutes) emit to: web UI toast, optional email via
  SMTP, optional webhook (Discord / Slack / ntfy / generic POST).
- Config additions: `alerts:` block with per-condition rules and a
  per-channel dispatch list.
- Surfaces a config-schema decision; couples naturally to either
  theme (hardware-breadth users need AIO-pump-stalled alerts;
  auth-rethink users need login-failure alerts).

## Packaging carryover

If any of these slipped v0.3:

- NixOS flake + module.
- Docker image + device-passthrough docs.
- AppImage, Flatpak — evaluate ROI after the above; may remain
  unfunded if `.deb` + `.rpm` + AUR + NixOS + Docker cover the
  overwhelming majority of the install surface.

## Test-coverage carryover

These were flagged in COVERAGE.md 2026-04-15 but did not make v0.3's
cut. Promote to v0.4 in-scope if coverage has not moved:

- `internal/hwmon/autoload.go` — sensors-detect parsing branches and
  module-probe loop untested. Deprioritised in v0.2.0 when probing
  moved to install time; still a gap.
- `internal/monitor` — 0 % coverage, reads temps and feeds the
  controller. `fs.FS`-injected tests plausibly land in an
  afternoon.
- `internal/nvidia` — 5 % coverage; NVML path behind purego is
  hard to test without either a real device or a mock harness.
  Budget a day.

## Out of scope (v0.4 explicit non-goals)

- Windows / macOS support. Linux-only remains the product promise.
- GUI desktop app (GTK / Qt / Electron). Web UI is the UI; a
  desktop wrapper is someone else's plugin.
- Telemetry-by-default. Strategy doc Tier 0.6 remains strictly
  opt-in; this does not change in v0.4.
- Clustering / fleet management. Out of scope indefinitely unless
  there is evidence of multi-host demand.

## Acceptance gates for tagging v0.4.0

Filled in when the theme is chosen. Candidate gates:

- If hardware breadth: at least one Corsair Commander device
  calibrates and runs end-to-end on a real AIO loop; existing
  hwmon + NVIDIA behaviour bit-for-bit unchanged on phoenix-MS-7D25
  and the home server; config migration round-trips v0.3 → v0.4 →
  v0.3 without edits.
- If auth/TLS rethink: existing single-admin password upgrades
  cleanly to the new users table; ACME issues against a public DNS
  name in a VM; audit log captures every mutating action in the
  UI smoke matrix.

## Competitive context (captured 2026-04-16)

- **CoolerControl** — daemon + built-in web UI, Rust/Vue, GPLv3,
  ~6k commits, 95 releases. Ships native packages for ~15 channels
  including AUR / NixOS / Gentoo / Unraid / Docker / AppImage.
  Supports USB AIOs, fan hubs, and storage-drive temps. Alerts
  built in. Direct competitor on architecture; the features above
  are the v0.4 delta that closes the gap.
- **fan2go** — Go daemon, YAML, TUI + Prometheus/REST, no web UI,
  AGPLv3, 334 stars. Not a direct competitor — different UX
  assumption. Worth tracking for its Prometheus exporter pattern
  if ventd adds metrics in v0.4.
- **nbfc-linux** — laptop-only (C port of NoteBook FanControl).
  Qt/Gtk frontends. Adjacent, not competing.
- **thinkfan**, **fancontrol** — baseline tools; ventd has left
  this class behind.
