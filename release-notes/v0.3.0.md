# ventd 0.3.0

**Released:** 2026-04-18
**Git tag:** `v0.3.0`
**Main SHA at tag:** `37d975d10deb7377a7da2562d2e577ecf8545e26`

---

## The short version

ventd 0.3.0 closes out Phase 1 of the roadmap: the architectural foundation that makes everything in Phase 2 and beyond possible. The daemon you download today looks and behaves almost identically to v0.2.x — same install, same dashboard, same safety. What changed is underneath: the code that talks to your fans has been reorganised so that in future releases, ventd can talk to entirely new classes of hardware (server BMCs, USB liquid coolers, laptop embedded controllers) without touching anything that already works.

If you're running v0.2.x, upgrade. There are no breaking changes.

## What shipped

### Hardware Abstraction Layer

Every code path that writes to fan hardware now goes through a single, well-tested interface called `FanBackend`. The existing hwmon (motherboard fans) and NVML (NVIDIA GPU fans) paths have been re-plumbed to use it. No behaviour changes — the same fans respond the same way. But Phase 2 backends (IPMI, AIO, laptop EC) can now be added as self-contained modules without touching the controller, the watchdog, or the calibration logic.

For users: invisible today, foundational for everything coming.

### Hot-loop performance

The core controller tick loop — the code that runs every two seconds on every machine ventd manages — has been reworked to eliminate per-tick memory allocations. Six specific optimisations:

- Sensor and smoothed-sensor maps are allocated once and reused, not recreated every tick.
- Compiled fan curves are cached, rebuilt only on config reload.
- Each tick reads the config snapshot once at the top, not repeatedly.
- Fan max-RPM is cached per fan at startup instead of being re-read every tick.
- The Points curve (piecewise-linear curves with many anchors) now uses binary search instead of linear scan.
- The Mix curve's intermediate slice is pooled, not allocated.

Net effect: about 50% less CPU for the same work on machines with many fans. On a typical desktop with 5 fans and 10 sensors, the difference is measurable but not life-changing; on a server with 30 fans and 50 sensors, it's the difference between ventd being unmeasurable and ventd showing up in `top`.

### Fingerprint-keyed hardware database

ventd now identifies your machine by a composite fingerprint (DMI board vendor/name, SMBIOS UUID category, chip IDs, PCI subsystem IDs) instead of matching hard-coded substrings. The initial database covers the three known boards from v0.2.x plus 15-plus additional motherboards, servers, and SBCs. Resolution order is exact → prefix → wildcard, logged at startup so you can see exactly why ventd classified your machine the way it did.

For users: better first-boot behaviour on more hardware. The database grows from here.

### Opt-in remote refresh for the hardware database

New command-line flag `--refresh-hwdb=<sha256>` fetches the latest `profiles.yaml` from the `ventd/hardware-profiles` repository, verifies it against a supplied SHA-256 pin, and merges it into the in-memory database. Local entries win on conflict. Off by default (`hwdb.allow_remote: false`); no startup network calls.

For users: opt-in flag. Fleet operators who want to pull new profiles without redeploying can; nobody else is affected.

### Runtime module loading without shell-out

The hwmon autoloader used to invoke `modinfo` as a subprocess for every unknown kernel module. That was slow (50ms per module), required `modinfo` to be installed (which fails on minimal distros), and created observability gaps (a `modinfo` invocation doesn't show up cleanly in structured logs). Now ventd parses `/lib/modules/$(uname -r)/modules.alias` and `modules.builtin-modinfo` directly in Go. Zero subprocess calls during module detection.

For users: faster startup on boards with many candidate modules, especially Alpine and other minimal distros where `modinfo` may not be installed at all.

### Module persistence is now append-not-overwrite

Running `ventd --probe-modules` twice on a dual-chip motherboard used to clobber the first run's result. Now the two chips' modules are merged and persisted together.

For users: boards with multiple Super I/O chips (some MSI, Gigabyte, ASRock server boards) no longer lose one of them on re-probe.

### Controller write-path resilience

When a PWM write fails (transient I/O error on a sysfs file, USB transient on future liquid backends), the controller now retries once at 50ms. If the retry also fails, ventd calls `watchdog.RestoreOne(path)` to hand control back to firmware auto for that specific fan, logs a structured `write_failed_restore_triggered` event, and moves on. Other fans are unaffected.

For users: one flaky fan doesn't disable fan control for the whole machine.

### Calibration via the HAL

Internal refactor of the calibration subsystem to drive fans through the new `FanBackend` interface instead of direct hwmon/NVML calls. Calibration behaviour is unchanged; the ZeroPWMSentinel safety net is unchanged. This was the last major subsystem still talking to hardware directly; now everything flows through one code path.

For users: invisible. Enables Phase 2 backends to inherit calibration for free.

### Watchdog coverage to 93 percent

The watchdog — the component that restores your fans to firmware control whenever ventd exits — was previously at 23 percent unit-test coverage. This release brings it to 93.6 percent, with seven safety invariants now bound 1:1 to named tests in `internal/watchdog/safety_test.go`. Every invariant has a rule file at `.claude/rules/watchdog-safety.md` that future changes can't silently violate.

For users: stronger guarantees that ventd will always clean up after itself, verified by tests that CI enforces.

### HAL contract invariants

Every backend (hwmon, NVML, and future Phase 2 backends) must pass `TestHAL_Contract`: eight invariants covering enumerate idempotence, read/write separation, write clamping, restore safety, capability stability, role determinism, and idempotent open. Enforced by the existing rulelint CI check.

For users: when Phase 2 backends land, they will arrive pre-audited against the same safety contract the existing backends obey.

### Security — Go toolchain bump + web UI headers

This release includes the Go 1.25.0 → 1.25.9 toolchain upgrade, which closes 17 Go standard library vulnerabilities reachable from ventd's production code paths (crypto/tls, crypto/x509, encoding/pem, net/http, net/url, os). `govulncheck` now reports zero reachable vulnerabilities.

Additionally, the web UI now sets `Permissions-Policy` headers that explicitly deny features ventd doesn't use (accelerometer, camera, geolocation, microphone, payment, USB). `Cross-Origin-Resource-Policy: same-origin` on HTML entry pages. ETag-based conditional GETs on the embedded UI assets with `Cache-Control: max-age=3600, must-revalidate`, so browsers cache aggressively but refresh on version bumps.

For users: stronger security posture. If you run ventd on a server reachable from untrusted networks, this is the upgrade that closes a stack of known stdlib CVEs.

### CI improvements

Two new lints run on every PR:

- **rulelint** (T0-META-01): scans `.claude/rules/*.md` and confirms every invariant rule is bound to a real test. Adding a rule without binding it, or removing a bound test without removing its rule, fails CI.
- **regresslint** (T0-META-02): scans closed bug-labelled GitHub issues and confirms each has either a matching `TestRegression_Issue<N>_*` test or a `no-regression-test` label. Keeps the regression-test discipline from decaying over time.

For users: invisible. Keeps the project honest.

## What didn't ship (but is coming)

See the [roadmap](../docs/roadmap.md). Summary:

- **Phase 2** — new backends for server BMCs (IPMI), AIO liquid coolers (Corsair/NZXT/Lian Li), laptop ECs (Framework/ThinkPad/Dell), ARM SBC PWM, and Apple Silicon via Asahi. Phase 2 work is already queued and starts immediately after this tag.
- **Phase 3** — install-time module loading discipline, coexistence with other fan daemons, `--dry-run` install mode.
- **Phase 4** — PI controller, autotuning, and optional MPC for genuinely quieter fans than a curve can achieve.
- **Phase 5** — in-use calibration (no 60-second howl ritual) and the hardware-profiles flywheel.
- **Phase 6** — Windows, macOS, FreeBSD, OpenBSD, illumos, Android.
- **Phase 7** — acoustic bearing-wear detection via microphone.

## Upgrading from v0.2.x

No breaking changes. No config migration needed. No user-visible behaviour changes.

```
curl -sSL https://raw.githubusercontent.com/ventd/ventd/main/scripts/install.sh | sudo bash
```

The installer replaces the binary, restarts the service, and you're on v0.3.0.

## Known issues

A structural audit of the codebase (ultrareview-1) surfaced a small number of non-blocking findings, none of which affect runtime safety or users. Each is tracked as a GitHub issue:

- **#265** — The `ventd/hardware-profiles` companion repo had a schema mismatch that caused `--refresh-hwdb` to silently return zero profiles. The schema has been corrected; the repo is intentionally empty until Phase 5 populates it.
- **#266** — A test in `internal/hal/contract_test.go` will panic when run on a host with a real NVIDIA GPU. Current CI has no GPU, so this is dormant. Will matter once a hardware-in-the-loop runner is wired in.
- **#267** — The `internal/hal` registry (the multi-backend composition layer) has no dedicated unit tests. Currently verified indirectly through backend tests; a direct test suite lands before Phase 2 Wave 1 dispatches.
- **#268** — Eight exported functions in `internal/hwmon` are unreachable from any caller and should be pruned or unexported.
- **#269** — The 54 HTTP endpoints in the web API have no external specification. Planned for pre-v0.5.
- **#271** — Twelve test-fixture packages share 33 lines of identical boilerplate; candidate for `go generate`-based codegen before Phase 2 adds more.
- **#272** — The manual-mode PWM write path in the controller doesn't yet have the same retry+RestoreOne treatment that the curve path received in this release. Scheduled for Phase 4.

None of these block production use. All are either pre-existing conditions identified by the audit, or scheduled for future phases.
