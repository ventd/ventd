# ventd 0.3.0 \u2014 release notes (draft)\n\n**Status:** Pre-draft. Not for publication. Final version lands in `release-notes/v0.3.0.md` at tag time.\n**Tag target:** v0.3.0\n**Tagging prerequisite:** ultrareview-1 returns clean OR all ultrareview blockers are resolved.\n**Expected tag date:** within 24 hours of ultrareview clean.\n\n---\n\n## The short version\n\nventd 0.3.0 closes out Phase 1 of the roadmap: the architectural foundation that makes everything in Phase 2 and beyond possible. The daemon you download today looks and behaves almost identically to v0.2.x \u2014 same install, same dashboard, same safety. What changed is underneath: the code that talks to your fans has been reorganised so that in future releases, ventd can talk to entirely new classes of hardware (server BMCs, USB liquid coolers, laptop embedded controllers) without touching anything that already works.\n\nIf you're running v0.2.x, upgrade. There are no breaking changes.\n\n## What shipped\n\n### Hardware Abstraction Layer\n\nEvery code path that writes to fan hardware now goes through a single, well-tested interface called `FanBackend`. The existing hwmon (motherboard fans) and NVML (NVIDIA GPU fans) paths have been re-plumbed to use it. No behaviour changes \u2014 the same fans respond the same way. But Phase 2 backends (IPMI, AIO, laptop EC) can now be added as self-contained modules without touching the controller, the watchdog, or the calibration logic.\n\nFor users: invisible today, foundational for everything coming.\n\n### Hot-loop performance\n\nThe core controller tick loop \u2014 the code that runs every two seconds on every machine ventd manages \u2014 has been reworked to eliminate per-tick memory allocations. Six specific optimisations:\n\n- Sensor and smoothed-sensor maps are allocated once and reused, not recreated every tick.\n- Compiled fan curves are cached, rebuilt only on config reload.\n- Each tick reads the config snapshot once at the top, not repeatedly.\n- Fan max-RPM is cached per fan at startup instead of being re-read every tick.\n- The Points curve (piecewise-linear curves with many anchors) now uses binary search instead of linear scan.\n- The Mix curve's intermediate slice is pooled, not allocated.\n\nNet effect: about 50% less CPU for the same work on machines with many fans. On a typical desktop with 5 fans and 10 sensors, the difference is measurable but not life-changing; on a server with 30 fans and 50 sensors, it's the difference between ventd being unmeasurable and ventd showing up in `top`.\n\n### Fingerprint-keyed hardware database\n\nventd now identifies your machine by a composite fingerprint (DMI board vendor/name, SMBIOS UUID category, chip IDs, PCI subsystem IDs) instead of matching hard-coded substrings. The initial database covers the three known boards from v0.2.x plus 15-plus additional motherboards, servers, and SBCs. Resolution order is exact \u2192 prefix \u2192 wildcard, logged at startup so you can see exactly why ventd classified your machine the way it did.\n\nFor users: better first-boot behaviour on more hardware. The database grows from here.\n\n### Opt-in remote refresh for the hardware database\n\nNew command-line flag `--refresh-hwdb=<sha256>` fetches the latest `profiles.yaml` from the `ventd/hardware-profiles` repository, verifies it against a supplied SHA-256 pin, and merges it into the in-memory database. Local entries win on conflict. Off by default (`hwdb.allow_remote: false`); no startup network calls.\n\nFor users: opt-in flag. Fleet operators who want to pull new profiles without redeploying can; nobody else is affected.\n\n### Runtime module loading without shell-out\n\nThe hwmon autoloader used to invoke `modinfo` as a subprocess for every unknown kernel module. That was slow (50ms per module), required `modinfo` to be installed (which fails on minimal distros), and created observability gaps (a `modinfo` invocation doesn't show up cleanly in structured logs). Now ventd parses `/lib/modules/$(uname -r)/modules.alias` and `modules.builtin-modinfo` directly in Go. Zero subprocess calls during module detection.\n\nFor users: faster startup on boards with many candidate modules, especially Alpine and other minimal distros where `modinfo` may not be installed at all.\n\n### Module persistence is now append-not-overwrite\n\nRunning `ventd --probe-modules` twice on a dual-chip motherboard used to clobber the first run's result. Now the two chips' modules are merged and persisted together.\n\nFor users: boards with multiple Super I/O chips (some MSI, Gigabyte, ASRock server boards) no longer lose one of them on re-probe.\n\n### Controller write-path resilience\n\nWhen a PWM write fails (transient I/O error on a sysfs file, USB transient on future liquid backends), the controller now retries once at 50ms. If the retry also fails, ventd calls `watchdog.RestoreOne(path)` to hand control back to firmware auto for that specific fan, logs a structured `write_failed_restore_triggered` event, and moves on. Other fans are unaffected.\n\nFor users: one flaky fan doesn't disable fan control for the whole machine.\n\n### Calibration via the HAL\n\nInternal refactor of the calibration subsystem to drive fans through the new `FanBackend` interface instead of direct hwmon/NVML calls. Calibration behaviour is unchanged; the ZeroPWMSentinel safety net is unchanged. This was the last major subsystem still talking to hardware directly; now everything flows through one code path.\n\nFor users: invisible. Enables Phase 2 backends to inherit calibration for free.\n\n### Watchdog coverage to 93 percent\n\nThe watchdog \u2014 the component that restores your fans to firmware control whenever ventd exits \u2014 was previously at 23 percent unit-test coverage. This release brings it to 93.6 percent, with seven safety invariants now bound 1:1 to named tests in `internal/watchdog/safety_test.go`. Every invariant has a rule file at `.claude/rules/watchdog-safety.md` that future changes can't silently violate.\n\nFor users: stronger guarantees that ventd will always clean up after itself, verified by tests that CI enforces.\n\n### HAL contract invariants\n\nEvery backend (hwmon, NVML, and future Phase 2 backends) must pass `TestHAL_Contract`: eight invariants covering enumerate idempotence, read/write separation, write clamping, restore safety, capability stability, role determinism, and idempotent open. Enforced by the existing rulelint CI check.\n\nFor users: when Phase 2 backends land, they will arrive pre-audited against the same safety contract the existing backends obey.\n\n### Security headers\n\nThe web UI now sets `Permissions-Policy` headers that explicitly deny features ventd doesn't use (accelerometer, camera, geolocation, microphone, payment, USB). `Cross-Origin-Resource-Policy: same-origin` on HTML entry pages. ETag-based conditional GETs on the embedded UI assets with `Cache-Control: max-age=3600, must-revalidate`, so browsers cache aggressively but refresh on version bumps.\n\nFor users: marginally better security posture. Not a vulnerability fix \u2014 no CVEs known \u2014 just defense in depth.\n\n### CI improvements\n\nTwo new lints run on every PR:\n\n- **rulelint** (T0-META-01): scans `.claude/rules/*.md` and confirms every invariant rule is bound to a real test. Adding a rule without binding it, or removing a bound test without removing its rule, fails CI.\n- **regresslint** (T0-META-02): scans closed bug-labelled GitHub issues and confirms each has either a matching `TestRegression_Issue<N>_*` test or a `no-regression-test` label. Keeps the regression-test discipline from decaying over time.\n\nFor users: invisible. Keeps the project honest.\n\n## What didn't ship (but is coming)\n\nSee the [roadmap](../docs/roadmap.md). Summary:\n\n- Phase 2 \u2014 new backends for server BMCs (IPMI), AIO liquid coolers (Corsair/NZXT/Lian Li), laptop ECs (Framework/ThinkPad/Dell), ARM SBC PWM, and Apple Silicon via Asahi. Phase 2 work is already queued and starts immediately after v0.3.0 is tagged.\n- Phase 3 \u2014 install-time module loading discipline, coexistence with other fan daemons, `--dry-run` install mode.\n- Phase 4 \u2014 PI controller, autotuning, and optional MPC for genuinely quieter fans than a curve can achieve.\n- Phase 5 \u2014 in-use calibration (no 60-second howl ritual) and the hardware-profiles flywheel.\n- Phase 6 \u2014 Windows, macOS, FreeBSD, OpenBSD, illumos, Android.\n- Phase 7 \u2014 acoustic bearing-wear detection via microphone.\n\n## Upgrading from v0.2.x\n\nNo breaking changes. No config migration needed. No user-visible behaviour changes.\n\n```\ncurl -sSL https://raw.githubusercontent.com/ventd/ventd/main/scripts/install.sh | sudo bash\n```\n\nThe installer replaces the binary, restarts the service, and you're on v0.3.0.\n\n## Known issues\n\n_To be filled in post-ultrareview.
