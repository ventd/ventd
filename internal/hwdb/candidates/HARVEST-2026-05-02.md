## How to read this report

Each entry below is a candidate `profiles-v1.yaml` block. Fields are filled to the extent that
public, authoritative data supports them; **uncertain fields are explicitly marked
`unknown`** rather than guessed. Phoenix should treat anything below `confidence: high`
as needing a HIL or community confirmation pass before it lands in `profiles-v1.yaml`.

Confidence rubric:

- **high** — Manufacturer manual published the fan layout, kernel hwmon driver source
  explicitly names the chip family, and a real Linux user has posted `sensors` output
  matching the layout. Two or more independent sources agree.
- **medium** — Manual covers the fan layout, chip identity has at least one community
  source (lm-sensors issue, Phoronix, Arch wiki) but no `sensors` paste, OR chip
  identity is inferred from the kernel driver's compatible-list and the manual.
- **low** — Fan layout known from the manual; chip family inferred from board
  generation/vendor pattern; no live `sensors` paste cited.

Schema notes:

- `dmi_board_vendor` and `dmi_board_name` come from `dmidecode -t baseboard`. Vendors are
  the literal strings — `ASUSTeK COMPUTER INC.`, `Micro-Star International Co., Ltd.`,
  `Gigabyte Technology Co., Ltd.`, `ASRock`, `Supermicro`, `Dell Inc.`, `HPE`, etc.
- `superio_chip` follows lm-sensors lowercase prefix style (`nct6798`, `it8689`, `nct6687`, etc.).
- `pwm_control` is the kernel module, not the chip name (e.g., `nct6775` covers all NCT677x/679x).
- `header_label` is the silkscreen label on the PCB — that is the user-facing name spec-12 wants.
- `max_pwm: 255` is the duty-cycle byte ceiling for hwmon `pwm_unit: duty_0_255` drivers.

## Gaps / next-time

Board families that need a future harvest pass — public sources were thin or
contradictory and I don't want fabricated entries:

1. **Lenovo Legion gaming laptops** (Legion 5 Pro, Legion Slim 7, etc.). The
   `thinkpad_acpi` driver covers ThinkPad business laptops well, but the Legion
   family uses a different EC firmware path. Phoronix mentioned a "New Lenovo
   Fan Driver" landing in Linux 7.1 — the harvest pass at the time of that
   driver merge will have the canonical board list and chip identifiers.
2. **Synology DSxxx+ / FSxxx** — Synology NAS fan control goes through
   `/dev/ttyS1` and proprietary `/dev/synobios` ioctl rather than hwmon. ventd
   would need a Synology-specific HAL backend that doesn't exist yet. The smallhacks
   blog from 2012 is the most authoritative public source and is dated.
   Recommend filing a tracking issue rather than guessing entries.
3. **QNAP TS-xxx** — Same problem class as Synology: QTS-managed serial protocol
   to a microcontroller, not hwmon. The TS-453D / TS-h973AX run on real x86 but
   QNAP firmware shadows the IT87 chip. TrueNAS-on-QNAP threads suggest
   community drivers exist but coverage is incomplete.
4. **Intel NUC11/12/13 EC fan path** — No upstream Linux driver. Some community
   work via `acpi_call`; not stable enough to ship as a recommended driver.
5. **Apple Silicon Mac mini (asahi)** — `applesmc` covers Intel Macs only; M1/M2
   thermal control is governed by AppleSMC over SPMI in asahi-linux. Out of scope
   for this v1 catalog but worth tracking.
6. **AMD AGESA boards from secondary brands (Biostar, Colorful, ECS)** — These
   ship NCT6775-family chips but the manuals are inconsistent on header
   silkscreen labels (some use SYSFAN1 with no underscore, some FAN_SYS_1).
   Need community submissions before catalog inclusion.
7. **MSI Z890 / B860 / Arrow Lake gen** — too new at harvest time; community
   `sensors` pastes pending. Expect another NCT6687D-class chip but layout may differ.
8. **ASUS B760M / Z690-M / Prime mATX line** — manuals exist but I prioritized
   ATX flagships; the mATX SKUs cut down on header counts and need their own
   per-SKU pass.
9. **Supermicro H13 / H14 (AMD EPYC Genoa / Turin)** — I covered X11/X12 Intel
   workstation only. EPYC servers have their own fan layout (typically 8 hot-swap
   carriers in 2U) and the BMC SDR enumeration differs.
10. **HPE iLO 6 / 7 (DL360 Gen11+)** — iLO 5 → 6 transition changed some IPMI
    command behaviour. RULE-IPMI-3 already refuses HPE writes at the controller
    level, so this is a low-priority gap, but a future "monitor-only" enrichment
    pass would benefit from confirmed Gen11/12 fan SDR names.
