Notes:
  R7 error 1 — docs/roadmap.md, Phase 3, second paragraph. The file says:
    "A zero-allocation ventd-recover binary, linked into the main binary
     as an embedded second executable, handles firmware auto-restore if
     the daemon gets stuck."
  Two faults. First, "linked into the main binary as an embedded second
  executable" invents a structural implementation detail. Masterplan
  P3-RECOVER-01 places it at cmd/ventd-recover/main.go — a separate
  binary with its own build — and the CC prompt "What to do" step 2
  explicitly forbade describing how the code is structured. Second,
  "handles firmware auto-restore if the daemon gets stuck" misstates
  purpose: the recover binary's role is restoring pwm_enable=1 after
  ungraceful daemon exit (SIGKILL, OOM kill, kernel panic) where the
  daemon process is gone. Stuck-daemon scenarios are the watchdog's
  domain, with a different restart path.

  R7 error 2 — docs/roadmap.md, Phase 10, second paragraph. The file
  says Permissions-Policy headers and strong ETag validation "prevent
  tampering or downgrade attacks." That is incorrect. Permissions-Policy
  is a browser feature-access denylist (camera, microphone, geolocation,
  USB, etc.); it does not prevent tampering or downgrade. ETag is an
  HTTP cache validator; it does not either. Anti-tampering and
  anti-downgrade are the domain of HSTS, signed artefacts, and
  subresource integrity, none of which P10-PERMPOL-01 claims to add.
  Incorrect public-facing security claims are a stronger failure than
  missing ones because they generate false assurance in readers.

  Scope of revision: rewrite only those two paragraphs. Everything
  else — structure, headings, word count, README link, CHANGELOG line,
  branch, PR shape, commit history — is clean and must be preserved.
