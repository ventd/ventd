# Feature ideas — parking lot for masterplan v2

**Status:** proposals only. Nothing here is committed to the masterplan. Review at each phase boundary; promote into the masterplan task catalogue when a feature passes three tests: (1) strategically aligned with `STRATEGY.md` wedges, (2) technically feasible given the HAL / architecture in place, (3) would move a measurable user-acquisition metric.

## How this document is used

Every Cowork session in downtime scans this file for features that have become newly feasible (phase completions unblock things) and for features that have become newly valuable (competitive landscape shifts raise their priority). At phase boundaries, the Cowork-orchestrator and the solo developer discuss which proposals to graduate into the masterplan.

Failed promotions don't leave the file \u2014 they move to the **Rejected** section at the bottom with a note explaining why, so future sessions don't re-propose them.

## Candidate feature proposals

### FP-01 — Per-application thermal profile hooks

**What:** When a high-load application launches (Blender render, video export, compile job, game), ventd silently switches to the user's \"heavy-load\" profile; when the application exits or idles, back to \"quiet.\" Detection via `systemd-cgtop` polling or eBPF on process-exec (preferred).

**Why:** The single biggest user complaint about fan-control tools is \"I want quiet most of the time but aggressive cooling when I actually need it.\" Curves can't do this \u2014 curves are reactive to temperature, not predictive of workload. Manual profile switching works but requires the user to remember. Automatic workload-driven switching is the thing nobody ships.

**Strategic fit:** Reinforces wedge #1 (zero-config first boot extends to zero-config runtime). Competitive gap: no Linux fan tool does this; FanControl.Releases has rudimentary \"auto switch profile on app X\" but it's closed-source.\n\n**Technical feasibility:** Phase 8+ territory. Requires (a) profile infrastructure from existing Phase 2 UI work \u2014 already shipped in v0.2.0. (b) process event detection \u2014 eBPF `sched_process_exec` tracepoint is clean; /proc polling is a fallback. (c) a simple rule DSL: `{ app: \"blender*\", profile: \"heavy\" }`.\n\n**Suggested masterplan slot:** Phase 8, new track `APP-PROFILES`.\n\n**Risk:** false-positive switching (switches to heavy when user is doing something innocuous that happens to touch a matched process name). Mitigation: require sustained load + app match, not just app launch.\n\n---\n\n### FP-02 — One-shot diagnostic report CLI\n\n**What:** `ventd --diag > report.txt`. Produces a single anonymised, pasteable report containing: DMI fingerprint (minus serial), hwmon tree with sensor labels, current calibration results, last 100 log lines, ventd version + git SHA. Drops PII by default; a flag re-includes serial/UUID for authorised debugging.\n\n**Why:** Issue triage is currently expensive. Users file bugs with partial info; maintainers ask three rounds of follow-up questions. A standardised diagnostic dump collapses this to one round-trip. Used by every serious open-source infra project (systemd-analyze dump, ss -s, sosreport).\n\n**Strategic fit:** Maintainer-productivity multiplier. Makes ventd's hardware-report issues (currently low-quality signal) into high-quality signal. Accelerates the Phase 5 profile-DB flywheel because every bug report is also a potential new profile entry.\n\n**Technical feasibility:** Low. Most of what it produces already exists inside the daemon; a CLI flag plus a gatherer function is a few hundred lines. Phase 8.\n\n**Suggested masterplan slot:** Phase 8, extends `P8-CLI-01`.\n\n**Risk:** accidental PII leakage. Mitigation: same anonymisation rules as P5-PROF-02 (the profile-capture task). Reuse that module.\n\n---\n\n### FP-03 — Acoustic budget mode\n\n**What:** User sets a maximum acoustic output (\"keep the machine below 35 dBA\"). The control loop \u2014 MPC or PI \u2014 treats this as a constraint: solve for minimum temperature subject to acoustic <= budget, instead of minimum RPM subject to temperature <= threshold.\n\n**Why:** Current fan-control paradigm is \"temperature first, noise incidentally.\" Users in sound-sensitive environments (home offices, recording setups, small-form-factor builds in living spaces) have the inverse priority: noise first, thermal throttling if necessary. No tool currently exposes this.\n\n**Strategic fit:** Extends wedge #8 (MPC learning controller). Creates a quantifiable marketing claim (\"stays under X dBA\") that reviewers can benchmark. Appeals directly to r/sffpc, the quiet-PC community, audio-workstation users.\n\n**Technical feasibility:** Phase 7+8 crossover. Requires Phase 7 acoustic baseline (so the system knows what each fan sounds like). MPC constraint is straightforward linear-program augmentation. Without acoustic baseline, degrades to \"RPM budget mode\" which is less accurate but still useful.\n\n**Suggested masterplan slot:** Phase 8, new track `ACOUSTIC-BUDGET`, depends on `P7-ACOUSTIC-01`.\n\n**Risk:** if thermal headroom is insufficient at the acoustic budget, hardware throttles. Mitigation: explicit user opt-in with a warning; publish a \"recommended minimum dBA for your hardware\" in the profile DB.\n\n---\n\n### FP-04 — Thermal event notifications\n\n**What:** User-configurable alerts: \"CPU held above 90\u00b0C for 20s,\" \"case fans at max for 10+ minutes,\" \"fan RPM dropped 30% from baseline\" (bearing wear precursor before Phase 7 acoustic detection ships). Delivered via desktop notifications, webhook, or email.\n\n**Why:** Thermal events today are silent until something fails. Users don't notice degraded cooling until thermal throttling becomes obvious (FPS drops in games, compile times increase). Early-warning turns passive monitoring into proactive maintenance. Enterprise fleet-management angle: Prometheus alerting integration is natural.\n\n**Strategic fit:** Reinforces wedge #7 (acoustic intelligence) by providing a non-acoustic early-warning path available from v0.6 instead of waiting for Phase 7 to ship acoustic work.\n\n**Technical feasibility:** Low. Trivial on top of the Prometheus metrics (P8-METRICS-01): user adds alerting rules themselves. Higher-value version: built-in alert definitions shipped with defaults, toggleable in the UI.\n\n**Suggested masterplan slot:** Phase 8, extends `P8-METRICS-01`.\n\n**Risk:** alert fatigue from overly sensitive defaults. Mitigation: conservative default thresholds, user can tighten.\n\n---\n\n### FP-05 — Per-sensor trust levels\n\n**What:** Some hardware sensors report values that are provably wrong (coretemp showing 100\u00b0C at idle on certain Intel SKUs, AMD tctl being offset by a fixed amount per CPU generation, BMCs with stuck-at values). ventd maintains a per-fingerprint trust map: \"on this board, sensor X's absolute value is unreliable but its derivative is,\" \"on that board, sensor Y must have a +20\u00b0C offset applied.\" UI surfaces this: sensors get a trust icon; untrusted sensors can't be used in curves unless the user explicitly overrides.\n\n**Why:** Current behavior on known-bad-sensor hardware is either (a) user doesn't notice, fan behavior is bizarre; or (b) user files a bug, maintainer investigates, finds it's a known hardware quirk, documents it manually. The profile DB makes this tractable: every known-bad sensor is a DB entry.\n\n**Strategic fit:** Direct extension of Phase 5's profile flywheel. Every profile capture can record sensor-trust observations. Strongly reinforces wedge #6 (profile-sourced first boot).\n\n**Technical feasibility:** Phase 5 extension. Schema for profiles.yaml adds a `sensor_quirks` section per fingerprint. Controller consults it when building curves.\n\n**Suggested masterplan slot:** Phase 5, extends `P5-PROF-01` or new `P5-TRUST-01`.\n\n**Risk:** false negatives \u2014 a sensor that IS accurate gets flagged. Mitigation: user can override trust with a warning.\n\n---\n\n### FP-06 — Config rollback (\"undo last change\")\n\n**What:** Every config Apply is automatically snapshotted with timestamp and change description. UI has a \"Rollback to...\" dropdown showing the last N applies. One-click restore. Safety net for users who change a curve and immediately see overheating.\n\n**Why:** Current failure mode: user edits a curve, Applies, realises it was wrong, has to reconstruct the previous curve from memory. Especially bad when the wrong curve caused thermal stress on components \u2014 they want to un-stress immediately, not debug what they changed.\n\n**Strategic fit:** Removes a perceived risk of using ventd's advanced features. \"Safe to experiment\" is a positive brand signal. Aligns with the autogenerated-everything philosophy \u2014 the system should remember so the user doesn't have to.\n\n**Technical feasibility:** Low-medium. Copy config.yaml to `/var/lib/ventd/history/config-<timestamp>.yaml` on every successful Apply. Rollback is just \"swap in old file, SIGHUP.\" Retain last 30 applies. Phase 8.\n\n**Suggested masterplan slot:** Phase 8 UX, new subtask on existing UX track.\n\n**Risk:** Unbounded growth of history dir. Mitigation: retention policy, default 30 or 7 days whichever is smaller.\n\n---\n\n### FP-07 — Import from CoolerControl / fancontrol\n\n**What:** `ventd --import-from coolercontrol` reads `~/.config/coolercontrol/config-ui.json` (or `/etc/fancontrol`), emits an equivalent `ventd.yaml`. One-way migration tool. \"You already used X? Here's your ventd config with one command.\"\n\n**Why:** Reduces the switching cost for users coming from competitors. \"Why would I switch?\" becomes \"because switching is free.\" Pulls directly from CoolerControl's 5-year install base.\n\n**Strategic fit:** Acquisition wedge. Pairs naturally with Phase 3 install improvements and Phase 6 cross-platform launch. Zero-friction migration becomes a story to tell in release posts.\n\n**Technical feasibility:** Medium. Need to understand the competitor config schemas well enough to translate. CoolerControl's schema is JSON and reasonably stable; fancontrol is bash-variable-ish and a parsing nuisance. Phase 3 is the natural home.\n\n**Suggested masterplan slot:** Phase 3, new `P3-IMPORT-01`.\n\n**Risk:** Competitors change config format; importer breaks. Mitigation: import is best-effort, produces a ventd config with comments indicating what didn't translate cleanly.\n\n---\n\n### FP-08 — Hardware-neutral API for remote fan control\n\n**What:** A small protocol over HTTPS (authenticated) that lets another ventd instance (or a fleet dashboard, or a third-party tool) control fans on this host. Same API the local web UI uses, but designed for cross-host access. Would feed the fleet-view work in `P8-FLEET-01` / `P8-FLEET-02`.\n\n**Why:** Homelab operators manage multiple machines. A single dashboard that controls fans on all of them is the natural ask. ventd's fleet features currently stop at discovery + link-to-UI; this would add centralised control.\n\n**Strategic fit:** Enterprise angle reinforces wedge #4 (IPMI enterprise/homelab market). Aligns with Phase 8 fleet work.\n\n**Technical feasibility:** Medium. `/api/v1/*` already exists. Adding auth suitable for cross-host requests (mTLS? Bearer token?) is the main design work.\n\n**Suggested masterplan slot:** Phase 8, extends `P8-FLEET-02`.\n\n**Risk:** Security model is harder than single-host. Tight mitigation: explicit enable flag, documented threat model, no defaults that would expose a new surface.\n\n---\n\n### FP-09 — Live terminal UI (`ventdctl top`)\n\n**What:** `ventdctl top` \u2014 full-screen terminal dashboard like `htop` but for fans and thermals. Live bar charts, fan/sensor list, recent log events. SSH-friendly.\n\n**Why:** Some users are headless. The web UI is mobile-friendly but the terminal-only user is underserved. A TUI wins that segment. Low effort given all the data is already flowing through SSE.\n\n**Strategic fit:** Reinforces \"ventd runs everywhere\" by including the workflow of users who never start a browser. r/selfhosted, r/homelab appreciate TUIs.\n\n**Technical feasibility:** Low. `bubbletea`-style TUI (pure Go, no CGO) subscribing to the existing SSE stream. Phase 8.\n\n**Suggested masterplan slot:** Phase 8, extends `P8-CLI-01`.\n\n**Risk:** new dependency (bubbletea). Mitigation: acceptable for an optional binary; main `ventd` binary unaffected.\n\n---\n\n### FP-10 — Fan \"personality\" profiles per user\n\n**What:** The profile system today is per-machine. This adds per-user: \"quiet (default),\" \"silent at night\" (overrides everything below 50\u00b0C), \"benchmark\" (max everything), \"live stream\" (holds all fans at a constant PWM to avoid noise fluctuation in a recording). Ships as profile templates the user can apply with one click from the wizard.\n\n**Why:** Fills the gap between \"I don't want to think about it\" and \"I want to tune every curve.\" Templates match real human use cases instead of asking users to design their own thermal strategy.\n\n**Strategic fit:** Makes onboarding smoother. Every template is a blog-post-able use case (\"How I set up ventd for streaming\").\n\n**Technical feasibility:** Low. Templates are just YAML snippets shipped in the binary. UI apply-template button already exists for profile-matched first boots.\n\n**Suggested masterplan slot:** Phase 9 UX.\n\n**Risk:** Templates won't fit every user. Mitigation: they're starting points, fully editable after apply.\n\n---\n\n## Rejected proposals (do not re-propose)\n\n_(Empty \u2014 populated as proposals are rejected with rationale.)_\n\n---\n\n## Meta-notes on this file\n\n- Every feature proposal includes: what, why, strategic fit, technical feasibility, suggested slot, risk.
