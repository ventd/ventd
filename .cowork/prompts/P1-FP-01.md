## What to do
1. **Create `internal/hwdb/fingerprint.go`** with the `HardwareFingerprint` struct. Fields (strings unless noted):
   - `BoardVendor` (from `/sys/class/dmi/id/board_vendor`)
   - `BoardName` (from `/sys/class/dmi/id/board_name`)
   - `BoardVersion` (from `/sys/class/dmi/id/board_version`)
   - `ProductFamily` (from `/sys/class/dmi/id/product_family`)
   - `SMBIOSUUID` — redacted to its first 8 hex digits then `xxxxxxxx...` form (never store a full UUID in the DB or in memory)
   - `ChipHexID` (from Super-I/O or SMBIOS chip-identification — hex string)
   - `PCISubsystem` (form `vvvv:dddd` for the primary PCI ID the board exposes)
   - `CPUMicrocode` (`/proc/cpuinfo` microcode field, e.g. `0x1000`)
   - JSON tags lowercase-with-underscore. `omitempty` on every field. Zero-value struct is a valid sentinel for "unknown".

2. **Read-side helpers** in the same package:
   - `func Read(sysfsRoot, procfsRoot string) (HardwareFingerprint, error)` — reads from real paths with injectable roots for tests; missing files are NOT errors, they leave the corresponding field empty.
   - Include a small `redactUUID(raw string) string` helper.

3. **Create `internal/hwdb/profiles.yaml`** embedded via `go:embed` in `internal/hwdb/db.go`. Schema:
   ```yaml
   version: 1
   profiles:
     - match:
         board_vendor: "Micro-Star International Co., Ltd."
         board_name: "MEG X570 UNIFY"
       modules: [nct6775]
       hints:
         primary_chip: nct6798
     - match:
         board_vendor: "ASUSTeK COMPUTER INC."
         board_name_prefix: "PRIME X670E"
       modules: [nct6775]
   ```
   Populate with:
   - The 3 entries currently in `autoload.go`'s hard-coded map (preserve exact behaviour for those boards).
   - At least 15 additional common boards you confidently know the DMI strings and fan-controller modules for:
     - MSI MAG/MEG X570, X670E, X870E, Z790, B650
     - ASUS ROG STRIX / PRIME / TUF (X570, X670E, X870E, Z790)
     - Gigabyte AORUS X570, X670E, Z790
     - ASRock X570, X670E, Z790
     - Supermicro X11, X12 (IPMI boards — flag these but leave module autoload empty since Phase 2 handles IPMI)
     - Dell Precision / OptiPlex (noted chip: nct6683, nct6687)
     - HP EliteDesk / ProDesk (noted chip: nct6683)
     - Framework 13 / 16 (EC-based; leave modules empty; hint EC)
     - Raspberry Pi 5 (PWM sysfs; leave modules empty)
   Do not guess module names you aren't confident about — omit uncertain entries rather than shipping a wrong module.

4. **Create `internal/hwdb/match.go`** with:
   - `func Match(fp HardwareFingerprint) (Profile, bool)` — returns the matched profile and `true`, or zero-value and `false`.
   - Resolution order, documented in a leading comment:
     1. Exact match on all non-empty match fields.
     2. Prefix match on `board_name_prefix` if set, exact on other non-empty fields.
     3. Wildcard: vendor-only match if no name-level matcher hit.
     4. If multiple profiles at the same tier match, return the most-specific (most match fields set) — deterministic tiebreak by lexical YAML order.

5. **Update `internal/hwmon/autoload.go`**:
   - Remove the hard-coded 3-entry map.
   - On boot, call `hwdb.Read` + `hwdb.Match`; if matched, use its `modules` for autoload.
   - On no match, log at INFO (not WARN) with the fingerprint (UUID redacted) so first-boot logs give the employer a ready-to-PR payload.

6. **Tests** at `internal/hwdb/match_test.go` — table-driven:
   - 18+ fingerprints against the DB, each asserting the expected profile hit.
   - Negative cases: completely unknown vendor, empty fingerprint, malformed YAML.

7. **CHANGELOG entry** under `## [Unreleased]` / `### Changed`:
   `- feat(hwdb): fingerprint-keyed hardware profile DB (≥18 boards); replaces 3-entry autoload map (#P1-FP-01)`
