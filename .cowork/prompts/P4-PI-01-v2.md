## Design — read carefully, do not deviate

### Sign convention correction (applied from PR #303 feedback)

The control law in this spec uses `err = sensor - Setpoint` (sensor
MINUS setpoint), NOT the inverted form. With `Kp > 0`:
- sensor > Setpoint → err > 0 → P > 0 → PWM increases → more cooling ✓
- sensor < Setpoint → err < 0 → P < 0 → PWM decreases → less cooling ✓

This is the direction the comments and YAML example both imply
(`"PWM per °C above setpoint"`), so all text in this spec uses that
convention.

### Interface extension

Add to `internal/curve/curve.go`:

```go
// StatefulCurve extends Curve for control laws that accumulate state across
// ticks (PI integral, MPC model residual, etc). The controller is responsible
// for persisting State per-channel and passing it back on each call.
//
// State is opaque to the controller; the curve returns a new state value
// (by value, not pointer) that the controller stores until the next tick.
// This keeps Evaluate callable from a single goroutine per channel without
// curve-side locking, and lets the curve impl choose its state layout.
//
// Implementations MUST be pure in (sensors, state) — same inputs produce
// the same (pwm, newState) — so a stale state value after a config reload
// simply retriggers convergence rather than diverging.
type StatefulCurve interface {
    Curve
    EvaluateStateful(sensors map[string]float64, state any, dtSeconds float64) (pwm uint8, newState any)
}
```

Do not make PICurve satisfy `Curve.Evaluate` via a lossy stub that pretends
integral state is zero; that would silently break anyone calling
`curve.Evaluate(sensors)` expecting it to do the right thing. Instead:

```go
func (c *PICurve) Evaluate(sensors map[string]float64) uint8 {
    pwm, _ := c.EvaluateStateful(sensors, PIState{}, 0)
    return pwm
}
```

Document clearly that calling `Evaluate` on a PICurve discards integral
action and returns only the proportional + feed-forward response (one
tick of it). This matches the fail-safe "stateless fallback" story and
makes diagnostic probes (e.g. a UI preview) sane.

### PICurve struct

`internal/curve/pi.go` (new):

```go
// PICurve is a proportional-integral controller that drives a fan PWM
// toward a temperature setpoint based on a single sensor input.
//
// Control law per tick:
//     err      = sensor - Setpoint                // POSITIVE when too hot
//     P        = Kp * err
//     I_new    = clamp(I_prev + Ki * err * dt, -IntegralClamp, +IntegralClamp)
//     output   = FeedForward + P + I_new          // all in 0..255 PWM units
//     pwm      = clampU8(output)                  // 0..255 before return
//
// Anti-windup: the integral clamp is applied BEFORE the output clamp,
// and the integral is NOT updated on ticks where the output clamps
// (back-calculation: if pwm saturates, roll back I_new to its pre-update
// value for this tick to prevent wind-up against a saturated actuator).
//
// Err-of-sign convention: higher temperature → higher PWM. Kp > 0, Ki >= 0.
// A config with Kp <= 0 or Ki < 0 is rejected at validate() time.
type PICurve struct {
    SensorName     string
    Setpoint       float64 // target temperature, °C
    Kp             float64 // proportional gain, PWM per °C
    Ki             float64 // integral gain, PWM per (°C·s)
    FeedForward    uint8   // baseline PWM added to P+I output
    IntegralClamp  float64 // max |I| in PWM units; hard anti-windup bound
}

// PIState is the per-channel integral carried across ticks.
type PIState struct {
    Integral float64 // accumulated integral term in PWM units
}
```

### Evaluate semantics — fail modes

All four conditions MUST return a safe PWM (255 — fail-high per existing
curve convention) AND reset Integral to 0 in the returned state:

1. `sensors` map missing the named sensor
2. Sensor value is NaN or ±Inf
3. `dtSeconds` is NaN, ±Inf, or negative
4. Resulting P+I arithmetic produces NaN/Inf before clamping

Log a structured event at WARN via `slog.Default()` on cases 1/2/4 (one
event per occurrence — per-tick debouncing is a future improvement, not
a v1 requirement).  Case 3 is a controller bug; log at ERROR.

### Integral clamp — exact shape

```go
// Inside EvaluateStateful, after computing err := sensor - Setpoint:
errContribution := c.Ki * err * dtSeconds
prev := s.Integral
candidate := prev + errContribution
if candidate > c.IntegralClamp {
    candidate = c.IntegralClamp
} else if candidate < -c.IntegralClamp {
    candidate = -c.IntegralClamp
}
// back-calculation: tentatively compute output; if it saturates, roll back
output := float64(c.FeedForward) + c.Kp*err + candidate
if output > 255 || output < 0 {
    // saturated — don't let integral grow in the saturation direction
    candidate = prev
    output = float64(c.FeedForward) + c.Kp*err + candidate
}
s.Integral = candidate
```

This is the industry-standard back-calc anti-windup, simplified for our
bounded 0..255 output. It must be verified by test, not by code review.

### Controller integration

In `internal/controller/controller.go`:

1. Add a `piState map[string]PIState` field to the `Controller` struct
   (key: channel ID string, value: current integral state). Initialize
   to empty map in the constructor.

2. Add `lastTickAt time.Time` field. Initialize to `time.Now()` in the
   constructor (so first-tick dt ≈ one poll interval, not `time.Since(zero time)` which would clamp).

3. In `tick()`, where the curve is evaluated for a channel:

```go
dt := time.Since(c.lastTickAt).Seconds()
if dt < 0.1 { dt = 0.1 }
if dt > 10.0 { dt = 10.0 }
c.lastTickAt = time.Now()

var pwm uint8
if sc, ok := curve.(curve.StatefulCurve); ok {
    prev := c.piState[channelID]
    p, newState := sc.EvaluateStateful(sensors, prev, dt)
    c.piState[channelID] = newState.(curve.PIState)
    pwm = p
} else {
    pwm = curve.Evaluate(sensors)
}
```

4. On config reload (SIGHUP path): purge `piState` for channels whose curve
   kind changed from/to `pi`. Channels whose curve stays PICurve across a
   reload keep their integral.

5. On panic-mode engagement: reset all piState entries to zero.
