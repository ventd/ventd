## Issues addressed

### #696 â€” atomicWrite *.tmp denied

`apparmor="DENIED" operation="mknod" name="/etc/ventd/config.yaml.tmp"`
appeared in dmesg on every wizard apply. The pre-fix v0.5.4 worktree
profile dropped the `/etc/ventd/*.tmp rw,` rule and added only
`/etc/ventd/*.tmp.* rw,`. Three writers in the codebase use plain
`<name>.tmp`:

- `internal/config/config.go:710` â€” config.yaml writes
- `internal/web/selfsigned.go:158` â€” TLS cert generation
- `internal/hwdb/capture.go:75` â€” pending hwdb captures

Three other writers use `<name>.tmp.<hex>`:

- `internal/state/state.go:83`
- `internal/web/authpersist/persist.go:95`
- `internal/calibrate/calibrate.go:1223`

**Resolution:** keep BOTH globs. Profile re-adds `/etc/ventd/*.tmp rw,`.

### #697 â€” hwmon[N] direct-attach paths denied

NVMe (hwmon1, hwmon2) and `thermal_zone0` (hwmon0) place the hwmon
directory directly under the device path with no intermediate
`hwmon/` segment. The pre-fix glob `/sys/devices/**/hwmon/**` requires
that literal segment and never matches direct-attach layouts.

**Resolution:** `/sys/devices/**/hwmon*/` glob (already on the v0.5.4
branch as `04d3a64`, carried into the merged fix).

### #694 â€” sd_notify(READY=1) never received by systemd

The original issue body diagnosed this as a Go-code bug
("`Notify(Ready)` only called from controller loop"). Source review
disproved that: `cmd/ventd/main.go:806` calls
`sdnotify.Notify(sdnotify.Ready)` outside the `if len(cfg.Controls) >
0` block, so it fires in setup-mode too.

The actual cause is in the AppArmor profile. dmesg shows:

```
apparmor="DENIED" operation="create" class="net"
info="failed protocol match" family="unix" sock_type="dgram"
```

`abstractions/base` provides `unix (create)` but not the
`(connect, send)` pair needed for the dgram path to
`/run/systemd/notify` (peer=unconfined). The profile's network rules
covered only `inet/inet6/netlink`, not `unix`.

Adding modern `unix (...) type=dgram,` rules alone was not enough.
The kernel kept emitting `class="net"` with "failed protocol match"
because without `abi <abi/4.0>,` at the top of the profile, the
parser's policy blob still uses legacy network-rule mediation for
AF_UNIX sockets â€” newer `unix` rules were silently inert.

**Resolution:**

```
abi <abi/4.0>,
...
profile ventd ... {
  ...
  unix (create, accept, bind, listen, connect, send, receive,
        getattr, getopt, setopt, shutdown),
}
```

Removing the operator-applied `hil-soak.conf` workaround
(`Type=simple`, `WatchdogSec=0`, `TimeoutStartSec=infinity`) and
restarting under stock `Type=notify`: daemon transitions to active
in <1s.

A defensive `Notify(sdnotify.Ready)` call was added in setup-mode
before the controllers branch so the daemon does not depend on a
single Notify path. sd_notify is idempotent.

### #699 â€” k10temp Tctl not surfaced as CPU sensor

Same root cause as #698. Once the hwmon read paths are not denied,
`internal/setup/setup.go:discoverCPUTempSensor` correctly identifies
the CPU sensor:

- Pass 1 iterates the `cpuChips` allowlist (includes `k10temp`).
- For each match, scans `temp*_input` and prefers labels containing
  `package`, `tdie`, or `tccd`.
- On this hardware: hwmon5 driver=k10temp; temp1_label=Tctl,
  temp3_label=Tccd1. Tccd1 matches "tccd" and is selected.

**Result:** `cpu_temp` sensor populated with
`/sys/class/hwmon/hwmon5/temp3_input` (k10temp/Tccd1) â€” the labeled
die-temperature sensor, semantically correct.

## Issues found during this session

### First-time-user blocker â€” empty config defaults to loopback bind

`internal/config/config.go:Empty()` returns `Listen: "127.0.0.1:9999"`
to satisfy RULE-INSTALL-03 (no plaintext bind on 0.0.0.0). On
first-boot the daemon generates a TLS cert and sets
`cfg.Web.TLSCert/TLSKey` â€” but never promotes `cfg.Web.Listen` from
the loopback default. Result: TLS is active but the listener stays
loopback-only. The LAN URL `install.sh` prints (built from
`hostname -I`) refuses connection from any other machine.

**Resolution:** `cmd/ventd/main.go` â€” after first-boot cert generation
succeeds, if `cfg.Web.Listen == "127.0.0.1:9999"` (the literal
`Empty()` default), promote to `0.0.0.0:9999`. Skip promotion if the
operator pre-provisioned a different value (do not override deliberate
config).

Verified: cold start logs `first-boot: promoting listen address to
wildcard now that TLS is active listen=0.0.0.0:9999` and emits the
correct LAN URL `https://192.168.7.10:9999`.

### Apparmor parser cannot represent `/usr/bin/[`

Adding `/usr/bin/[ rix,` produced
`apparmor_parser: Regex grouping error: Unclosed grouping or character
class`. The bare `[` is interpreted as the start of a regex char
class. `{[}`, `\[`, and `[[]` all fail the same way.

The shipped `ventd-wait-hwmon` script uses `test`, not `[`, so the
omission is a no-op today. Mitigation: any future shell script under
`AppArmorProfile=ventd` must use `test` instead of `[`. A grep guard
in CI would catch regressions:

```
grep -nE '\b\[ ' deploy/*.sh scripts/*.sh
```

### `ConfigurationDirectory` mode warning every restart

`journalctl -u ventd` shows on every restart:

```
ventd.service: ConfigurationDirectory 'ventd' already exists but the
mode is different. (File system: 770 ConfigurationDirectoryMode: 750)
```

The `ConfigurationDirectory=ventd` directive expects mode 0750, but
the install gives `/etc/ventd/` mode 0770. Cosmetic on first boot
(the daemon works), but it is a warning a first-time user sees in
`systemctl status` and may panic about. Reconcile install.sh chmod
or the systemd directive.
